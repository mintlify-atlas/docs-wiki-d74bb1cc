The ABP Framework repository is a multi-stack monorepo: a .NET runtime, a parallel set of business modules, an Angular UI workspace, opinionated solution templates, ABP Studio source-code packages, and the build/deploy/tooling glue that ships them as NuGet and npm artifacts. This page is the map. It walks every top-level directory you will see in a fresh clone of `abpframework/abp`, then drills into the three biggest roots — `framework/`, `modules/`, and `templates/` — with per-area tables so you can locate any subsystem by name.

<Note>
  The root `Directory.Build.props` is intentionally tiny (it only flips the `IsTestProject` flag for `coverlet.collector`), so most build configuration lives in the per-area `common.props` files and in `Directory.Packages.props` for central package versioning.
</Note>

## Top-level directory map

Run `ls` at the repo root and you will see the following entries. Every directory is documented below, with a one-line "what's inside" plus the typical artifact it produces.

| Directory | Contents | Output / role |
| --- | --- | --- |
| `framework/` | All `Volo.Abp.*` runtime packages — DI core, modularity, DDD, EF Core, MongoDB, Mvc, Blazor, Auth, MultiTenancy, EventBus, BackgroundJobs, etc. (≈150 projects under `framework/src/`). | NuGet packages on nuget.org. |
| `modules/` | Pre-built ABP application modules (Identity, OpenIddict, Account, Tenant Management, Audit Logging, Feature Management, Setting Management, Permission Management, BLOB Storing DB, Background Jobs, CMS Kit, Docs, Blogging, Basic Theme, Virtual File Explorer, Users, Client Simulation). | NuGet packages prefixed `Volo.Abp.*`. |
| `templates/` | Solution starter templates: `app` (full layered), `app-nolayers` (single project), `module`, `console`, `maui`, `wpf`. | ZIP-packed templates consumed by `abp new` and ABP Studio. |
| `npm/` | The Angular / web client workspace (lerna-managed under `npm/ng-packs/packages/`), plus the React Native packs and publish scripts. | npm packages under `@abp/*`. |
| `apiSpec/` | OpenAPI / Mintlify-style markdown specs (e.g. `Microsoft_AspNetCore_Routing_AbpEndpointRouterOptions.md`). | Documentation source. |
| `build/` | PowerShell scripts (`build-all.ps1`, `build-all-release.ps1`, `test-all.ps1`, `common.ps1`) used by CI. | Build orchestration. |
| `deploy/` | Numbered release scripts (`1-fetch-and-build.ps1` through `8-download-release-zip.ps1`) plus NuGet/npm publishing and GitHub-release helpers. | Release pipeline. |
| `docs/` | Versioned documentation tree (`docs/en/`, `docs/es/`, `docs/zh-Hans/`, `docs/cs/`, `docs/pt-BR/`, plus `docs-langs.json`). | Source for docs.abp.io. |
| `source-code/` | Solution (`SourceCodes.sln`) wrapping `*.SourceCode` versions of each module so users can replace the NuGet reference with a project reference. | "Replace a module with its source" workflow. |
| `studio/` | ABP Studio-specific bundles. Currently `studio/source-codes/` mirroring the `source-code/` set for the Studio's "Get source code" UX. | Studio integration. |
| `test/` | Cross-cutting test apps: `test/AbpPerfTest/` (performance harness) and `test/DistEvents/` (distributed event bus end-to-end). | Manual / soak tests. |
| `tools/` | Auxiliary CLI tools: `tools/github-changelog-generator/`, `tools/localization-key-synchronizer/`, `tools/nuget/`. | Maintenance tooling. |
| `abp_io/` | `AbpIoLocalization` resource project for the abp.io marketing site translations. | Localization JSON. |
| `nupkg/` | Local NuGet packing workspace: `pack.ps1`, `push_packages.ps1`, `push-nightly-packages-myget.ps1`, `unit_test.ps1`. | Local pack/push scripts. |

There are also the standard root files:

| File | Purpose |
| --- | --- |
| `Directory.Build.props` | Global MSBuild props — only flips `IsTestProject` and adds `coverlet.collector` to test projects. |
| `Directory.Packages.props` | Central package version management (`ManagePackageVersionsCentrally=true`) listing every third-party `PackageVersion`. |
| `global.json` | Pins the .NET SDK: `"version": "8.0.100"`, `"rollForward": "latestFeature"`. |
| `common.props`, `common.test.props`, `configureawait.props` | Shared compiler / analyzer settings imported by area-specific `.csproj` files. |
| `NuGet.Config` | Adds the upstream nuget.org feed and the MyGet nightly feed. |
| `README.md` | Top-level project description (also rendered on the NuGet listing for `Volo.Abp.Core`). |
| `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, `LICENSE.md` | Standard governance files. |
| `latest-versions.json` | Snapshot of the current published version per package, used by tooling. |
| `delete-bin-obj.ps1` | Convenience cleanup script. |

## `framework/src/` — the runtime fabric

`framework/src/` contains the `Volo.Abp.*` projects that compile into the framework's NuGet packages. There are far too many to spell out individually here — the [Architecture](/overview/architecture) page maps the headline concerns to their packages — but they group into a small number of families.

| Family | Pattern | Representative packages |
| --- | --- | --- |
| Core kernel | `Volo.Abp.Core` | `Volo.Abp.Core` (AbpApplicationBase, AbpModule, DI conventions, Modularity loader, Threading, Guids, Timing, Internal extension methods). |
| DDD building blocks | `Volo.Abp.Ddd.*` | `Volo.Abp.Ddd.Domain.Shared`, `Volo.Abp.Ddd.Domain`, `Volo.Abp.Ddd.Application.Contracts`, `Volo.Abp.Ddd.Application`. |
| Data & persistence | `Volo.Abp.EntityFrameworkCore*`, `Volo.Abp.MongoDB`, `Volo.Abp.Dapper`, `Volo.Abp.MemoryDb`, `Volo.Abp.Data`, `Volo.Abp.Uow` | DB provider matrix + Unit of Work primitives. |
| Cross-cutting infra | `Volo.Abp.{Auditing,Authorization,Caching,Localization,Validation,ObjectMapping,ObjectExtending,Settings,Features,GlobalFeatures,Guids,Threading,Timing,Specifications,Json*,Serialization,Security,ExceptionHandling,Emailing,Sms,Imaging.*,TextTemplating.*,Minify}` | Per-concern slices following the pattern `<Concern>` + `<Concern>.Abstractions`. |
| Messaging | `Volo.Abp.EventBus`, `Volo.Abp.EventBus.Abstractions`, `Volo.Abp.EventBus.{RabbitMQ,Kafka,Azure,Dapr,Rebus}`, `Volo.Abp.{RabbitMQ,AzureServiceBus,Kafka,Dapr}` | Local + distributed event bus and transport primitives. |
| Background work | `Volo.Abp.BackgroundJobs[.Abstractions/.HangFire/.Quartz/.RabbitMQ]`, `Volo.Abp.BackgroundWorkers[.Hangfire/.Quartz]`, `Volo.Abp.Quartz`, `Volo.Abp.HangFire` | Fire-and-forget jobs and periodic workers. |
| Multi-tenancy | `Volo.Abp.MultiTenancy`, `Volo.Abp.MultiTenancy.Abstractions`, `Volo.Abp.AspNetCore.MultiTenancy`, `Volo.Abp.AspNetCore.Mvc.UI.MultiTenancy` | Tenant resolution + UI. |
| BLOB storing | `Volo.Abp.BlobStoring`, `Volo.Abp.BlobStoring.{Azure,Aws,Minio,FileSystem,Aliyun}` | Provider matrix. |
| HTTP surface | `Volo.Abp.Http`, `Volo.Abp.Http.Abstractions`, `Volo.Abp.Http.Client`, `Volo.Abp.Http.Client.{IdentityModel,IdentityModel.Web,IdentityModel.WebAssembly,IdentityModel.MauiBlazor,Dapr,Web}`, `Volo.Abp.RemoteServices` | API description model + dynamic client proxies. |
| ASP.NET Core integration | `Volo.Abp.AspNetCore`, `Volo.Abp.AspNetCore.{Authentication.JwtBearer,Authentication.OAuth,Authentication.OpenIdConnect,Serilog,SignalR,TestBase}` | Host integration. |
| MVC & Razor UI | `Volo.Abp.AspNetCore.Mvc[.Contracts/.Client/.Client.Common/.Dapr/.Dapr.EventBus/.NewtonsoftJson/.UI/.UI.Bootstrap/.UI.Bundling/.UI.Bundling.Abstractions/.UI.MultiTenancy/.UI.Packages/.UI.Theme.Shared/.UI.Theme.Shared.Demo/.UI.Widgets]` | The MVC / Razor Pages stack with the Bootstrap-based "shared theme". |
| Blazor / MAUI | `Volo.Abp.AspNetCore.Components[.Server/.Server.Theming/.Web/.Web.Theming/.WebAssembly/.WebAssembly.Theming/.MauiBlazor/.MauiBlazor.Theming]`, `Volo.Abp.BlazoriseUI`, `Volo.Abp.Maui.Client`, `Volo.Abp.Autofac.WebAssembly` | Component stack + Blazorise wrapper. |
| Container & interception | `Volo.Abp.Autofac`, `Volo.Abp.Castle.Core` | Autofac integration and the Castle DynamicProxy bridge. |
| Distributed locking | `Volo.Abp.DistributedLocking`, `Volo.Abp.DistributedLocking.Abstractions`, `Volo.Abp.DistributedLocking.Dapr` | `IAbpDistributedLock` + providers. |
| Identity / auth helpers | `Volo.Abp.Authorization`, `Volo.Abp.Authorization.Abstractions`, `Volo.Abp.Security`, `Volo.Abp.IdentityModel`, `Volo.Abp.Ldap`, `Volo.Abp.Ldap.Abstractions`, `Volo.Abp.Gdpr.Abstractions` | Permission system and token / LDAP / GDPR helpers. |
| CLI | `Volo.Abp.Cli`, `Volo.Abp.Cli.Core` | The `abp` global tool. |
| Test base | `Volo.Abp.TestBase`, `Volo.Abp.AspNetCore.TestBase` | xUnit + Shouldly + Autofac scaffolding for tests. |
| Misc | `Volo.Abp.UI`, `Volo.Abp.UI.Navigation`, `Volo.Abp.MultiLingualObjects`, `Volo.Abp.ApiVersioning.Abstractions`, `Volo.Abp.Swashbuckle`, `Volo.Abp.VirtualFileSystem`, `Volo.Abp.MailKit`, `Volo.Abp.FluentValidation`, `Volo.Abp.AutoMapper` | One-off bridges & helpers. |

<Tip>
  The naming is rigorously consistent: every concern that has a public contract surface ships a `*.Abstractions` package alongside its implementation package (e.g. `Volo.Abp.EventBus.Abstractions` vs `Volo.Abp.EventBus`). Reference the abstractions in domain code and the implementation in the composition root.
</Tip>

## `modules/` — application modules

Each subdirectory under `modules/` is a self-contained ABP application module with its own `src/`, `test/`, and `.sln` file. The modules ship as NuGet packages (and Angular packages where they have UI).

| Directory | One-line role |
| --- | --- |
| `modules/account` | Login / register / external auth UI on top of Identity. |
| `modules/audit-logging` | Persisted audit log store (action-level, entity-change-level). |
| `modules/background-jobs` | Persistent store implementation of `IBackgroundJobStore` (uses EF Core / MongoDB). |
| `modules/basic-theme` | The default ABP Bootstrap theme (MVC + Blazor + Angular variants). |
| `modules/blob-storing-database` | Database-backed `IBlobContainer` provider used as the default in templates. |
| `modules/blogging` | Lightweight blogging engine (legacy/community module). |
| `modules/client-simulation` | Load / scenario simulation harness for testing apps. |
| `modules/cms-kit` | Drop-in CMS primitives: pages, menus, blogs, comments, tags, reactions. |
| `modules/docs` | Markdown-driven documentation module (powers docs.abp.io itself). |
| `modules/feature-management` | Persisted, scoped store for `FeatureDefinition` values. |
| `modules/identity` | Users / roles / claims / OUs / sessions on top of Microsoft.AspNetCore.Identity. |
| `modules/identityserver` | Legacy IdentityServer4 integration (preserved for upgrade paths). |
| `modules/openiddict` | Current OpenID Connect server using OpenIddict 5. |
| `modules/permission-management` | Persisted permission grant store and UI. |
| `modules/setting-management` | Persisted setting values store + admin UI for `SettingDefinition`s. |
| `modules/tenant-management` | Tenant CRUD + connection-string management for multi-tenant SaaS. |
| `modules/users` | Shared user-facing primitives (`IExternalUserLookupService`, profile DTOs). |
| `modules/virtual-file-explorer` | Dev-time UI to browse the in-memory virtual file system. |

## `templates/` — solution scaffolds

The `abp new` command and ABP Studio use `templates/` to scaffold customer solutions. Every template lives in its own subdirectory and is zipped by `templates/zip-templates.ps1` before publishing.

| Directory | Stacks | Description |
| --- | --- | --- |
| `templates/app` | `aspnet-core/`, `angular/`, `react-native/` | Full DDD-layered web solution: separate `Domain.Shared`, `Domain`, `Application.Contracts`, `Application`, `HttpApi`, `HttpApi.Client`, `EntityFrameworkCore`, `MongoDB`, `Web`/`Blazor`/`Blazor.Server`/`Blazor.Server.Tiered`, `HttpApi.Host`, `HttpApi.HostWithIds`, `AuthServer`, `DbMigrator`, `Web.Host`. |
| `templates/app-nolayers` | `aspnet-core/`, `angular/` | Single-project counterpart — same features without the layer split, ideal for small services. |
| `templates/module` | `aspnet-core/`, `angular/` | Scaffold for building your own application module that follows the framework's layering conventions. |
| `templates/console` | `aspnet-core/` (Generic Host) | Headless console host using `IHost` + `IAbpApplicationWithExternalServiceProvider`. |
| `templates/maui` | `aspnet-core/` | .NET MAUI client wired through `Volo.Abp.Maui.Client` and the IdentityModel MAUI access-token helper. |
| `templates/wpf` | `aspnet-core/` | WPF desktop client referencing the shared `HttpApi.Client`. |

The root `templates/` directory also holds `Directory.Packages.props` (separate from the framework's own central package list — templates have their own) and `.yarnrc` for the Angular variants.

## `build/`, `deploy/`, `nupkg/` — release pipeline

Three directories cooperate to produce official releases.

<Steps>
  <Step title="build/ — compile & test">
    `build/build-all.ps1` (debug) and `build/build-all-release.ps1` (release) walk every `.sln`. `build/test-all.ps1` runs xUnit across the repo. `build/common.ps1` carries shared helpers.
  </Step>
  <Step title="nupkg/ — local pack">
    `nupkg/pack.ps1` packs every Release-configured project into `.nupkg` files. `nupkg/unit_test.ps1` runs tests, `nupkg/push_packages.ps1` and `nupkg/push-nightly-packages-myget.ps1` push to nuget.org and MyGet respectively.
  </Step>
  <Step title="deploy/ — orchestrated release">
    The numbered scripts run in sequence: `1-fetch-and-build.ps1`, `2-nuget-pack.ps1`, `3-nuget-push.ps1`, `4-npm-publish-mvc.ps1`, `5-npm-publish-angular.ps1`, `6-git-commit.ps1`, `7-publish-github-release.ps1`, `8-download-release-zip.ps1`. `_run_all.ps1` invokes them all. `new-github-release-function.psm1` carries the GitHub release helper.
  </Step>
</Steps>

## `docs/` and `apiSpec/`

`docs/` is the source for docs.abp.io. It is organised by language: `docs/en/`, `docs/es/`, `docs/zh-Hans/`, `docs/cs/`, `docs/pt-BR/`, with `docs/docs-langs.json` enumerating the registered locales. `apiSpec/` carries hand-curated API spec markdown (currently `Microsoft_AspNetCore_Routing_AbpEndpointRouterOptions.md`) used to seed the public reference pages.

## `framework/test/` — runtime test scaffolding

Adjacent to `framework/src/` is `framework/test/`, which contains the xUnit + Shouldly test projects that prove the framework's own behaviour. Tests are grouped by package — `framework/test/Volo.Abp.Core.Tests/`, `framework/test/Volo.Abp.Autofac.Tests/`, `framework/test/Volo.Abp.Ddd.Domain.Tests/`, and so on — and they share the bases from `framework/src/Volo.Abp.TestBase/` and `framework/src/Volo.Abp.AspNetCore.TestBase/`. This is also where you will find sample modules (`AbpTestApp.Domain`, `AbpTestApp.Application`) used as in-test fixtures.

## `modules/` — typical project layout per module

Most directories under `modules/` follow the same internal layout — an aligned `src/` and `test/` plus a top-level `.sln`, `.abpmdl`, `.abpsln` and `NuGet.md`. The `.abpmdl` file is the ABP Studio module manifest (which projects participate, which layers are replaceable); the `.abpsln` file is a Studio solution definition. Engineers usually open `*.sln` for VS / Rider, and Studio reads the other two.

A representative shape (drawn from `modules/identity/`):

| File / folder | Role |
| --- | --- |
| `modules/identity/src/` | The 15 layered projects shown earlier in this page. |
| `modules/identity/test/` | xUnit projects exercising domain, application, EF Core and Mongo providers. |
| `modules/identity/Volo.Abp.Identity.sln` | Visual Studio / Rider solution. |
| `modules/identity/Volo.Abp.Identity.abpsln` | ABP Studio solution. |
| `modules/identity/Volo.Abp.Identity.abpmdl` | ABP Studio module manifest (declares which layers can be replaced with source). |
| `modules/identity/NuGet.md` | The NuGet listing description for the package. |
| `modules/identity/README.md` | Repo-side description, links to docs.abp.io. |

A few modules deviate slightly — `modules/cms-kit/` adds public-facing sub-modules, `modules/openiddict/` includes an `Installer` project for the post-install OpenIddict app seeding, and `modules/docs/` carries Markdown content alongside the code — but the layered project pattern is universal.

## `npm/ng-packs/packs/` — pack staging

Alongside `npm/ng-packs/packages/`, there is `npm/ng-packs/packs/` — the local tarball staging directory used by `npm/preview-publish.ps1`. CI writes packed `*.tgz` archives here before they are pushed to npm via `npm/publish-ng.ps1`.

## `npm/` root scripts

The `npm/` root contains a handful of orchestration scripts that compose the per-package publishing:

| File | Purpose |
| --- | --- |
| `npm/lerna.json` | Lerna config — fixed versioning across the workspace. |
| `npm/package.json` | Workspace root manifest with cross-package dev dependencies. |
| `npm/package-update-script.js` | Script used by maintainers to bump every `@abp/*` reference simultaneously. |
| `npm/publish-mvc.ps1` | Publishes the MVC-side JS packages. |
| `npm/publish-ng.ps1` | Publishes the Angular packages. |
| `npm/preview-publish.ps1` | Dry-runs the publish pipeline. |
| `npm/publish-utils.js` | Shared Node helpers used by the publish scripts. |
