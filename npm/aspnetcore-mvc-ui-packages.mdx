`npm/packs/` is the Lerna monorepo that produces the `@abp/*` runtime packages installed by every ABP ASP.NET Core MVC/Razor Pages application. The packs themselves are deliberately tiny — most are vendor wrappers that pull in a battle-tested library (jQuery, Bootstrap, DataTables.net, SweetAlert2, …) and ship a single `abp.resourcemapping.js` describing which physical files should be copied into the host application's `wwwroot/libs/` folder. A small number of ABP-authored packs — `@abp/core`, `@abp/utils`, `@abp/aspnetcore.mvc.ui` and the `@abp/aspnetcore.mvc.ui.theme.*` aggregators — sit on top of those vendor wrappers and provide the JavaScript runtime (`abp.js`), the gulp helper that runs the copy task, and the dependency-bundle meta-packages that themes import.

This page documents the actual shape of `npm/packs/`, how a typical pack is structured, what `abp.resourcemapping.js` files contain, and how the gulp pipeline turns that metadata into `wwwroot/libs/`.

## The `abp.resourcemapping.js` convention

Every pack ships an optional `abp.resourcemapping.js` file with two top-level keys: `aliases` and `mappings`. The aliases are reusable placeholders that resolve at runtime; the mappings are `glob → destination` pairs that drive the gulp `copyResourcesTask`.

The defaults (applied by `copy-resources.js` when a host project does not override them):

```js packs/aspnetcore.mvc.ui/gulp/copy-resources.js (excerpt)
function normalizeResourceMapping(resourcemapping) {
    var defaultSettings = {
        aliases: {
            "@node_modules": "./node_modules",
            "@libs": "./wwwroot/libs"
        },
        clean: [
            "@libs"
        ]
    };

    extendObject(defaultSettings.aliases, resourcemapping.aliases);
    resourcemapping.aliases = defaultSettings.aliases;

    resourcemapping.clean = resourcemapping.clean || defaultSettings.clean;

    return resourcemapping;
}
```

So `@node_modules/...` resolves to `./node_modules/...` and `@libs/...` resolves to `./wwwroot/libs/...` inside the host application. A handful of real-world mappings:

### How transitive mappings are resolved

`copy-resources.js` does not just read the consumer's own `abp.resourcemapping.js` — it walks every entry in `dependencies` recursively and merges their mappings. The relevant code:

```js packs/aspnetcore.mvc.ui/gulp/copy-resources.js (excerpt)
function buildResourceMapping(packagePath) {
    if (investigatedPackagePaths[packagePath]) {
        return {};
    }
    investigatedPackagePaths[packagePath] = 'OK';

    var packageJson = requireOptional(path.join(packagePath, 'package.json'));
    var resourcemapping =
        requireOptional(path.join(packagePath, 'abp.resourcemapping.js')) || {};

    if (packageJson && packageJson.dependencies) {
        var aliases = {};
        var mappings = {};

        for (var dependency in packageJson.dependencies) {
            if (packageJson.dependencies.hasOwnProperty(dependency)) {
                var dependedPackagePath =
                    path.join(rootPath, 'node_modules', dependency);
                var importedResourceMapping =
                    buildResourceMapping(dependedPackagePath);
                extendObject(aliases, importedResourceMapping.aliases);
                extendObject(mappings, importedResourceMapping.mappings);
            }
        }

        extendObject(aliases, resourcemapping.aliases);
        extendObject(mappings, resourcemapping.mappings);

        resourcemapping.aliases = aliases;
        resourcemapping.mappings = mappings;
    }

    return resourcemapping;
}
```

This is why installing `@abp/aspnetcore.mvc.ui.theme.shared` is enough — the recursion picks up the mappings from every transitive vendor pack (`bootstrap`, `font-awesome`, `jquery-form`, …) and copies the full library set in a single `yarn gulp` invocation.

The task itself is just a `gulp.src().pipe(gulp.dest())` per mapping:

```js packs/aspnetcore.mvc.ui/gulp/copy-resources.js
function copyResourcesTask (path) {
    rootPath = path;
    resourceMapping = normalizeResourceMapping(buildResourceMapping(rootPath));

    cleanDirsAndFiles(resourceMapping.clean);

    var tasks = [];

    if (resourceMapping.mappings) {
        for (var mapping in resourceMapping.mappings) {
            if (resourceMapping.mappings.hasOwnProperty(mapping)) {
                var destination = replaceAliases(resourceMapping.mappings[mapping]);
                if (fs.existsSync(destination)) continue;

                var source = replaceAliases(mapping);
                tasks.push(
                    gulp.src(source).pipe(gulp.dest(destination))
                );
            }
        }
    }

    return merge(tasks);
}

module.exports = copyResourcesTask;
```

A host project's `gulpfile.js` only needs to require this and call it with `__dirname`.

## Dependency layering

The packs form a shallow DAG. `@abp/utils` is the leaf; `@abp/core` depends on it; every vendor wrapper depends on `@abp/core`; the two theme aggregators sit on top.

```mermaid
flowchart LR
  utils[@abp/utils] --> core[@abp/core]
  core --> jquery[@abp/jquery]
  core --> bootstrap[@abp/bootstrap]
  core --> select2[@abp/select2]
  core --> dt[@abp/datatables.net]
  dt --> dtbs5[@abp/datatables.net-bs5]
  bootstrap --> shared[@abp/aspnetcore.mvc.ui.theme.shared]
  jquery --> shared
  dtbs5 --> shared
  fa[@abp/font-awesome] --> shared
  swal[@abp/sweetalert2] --> shared
  toastr[@abp/toastr] --> shared
  mvcui[@abp/aspnetcore.mvc.ui] --> shared
  shared --> basic[@abp/aspnetcore.mvc.ui.theme.basic]
```

Because the gulp task walks `dependencies` recursively, depending on `@abp/aspnetcore.mvc.ui.theme.shared` is enough to materialise every box on the diagram into `wwwroot/libs/`.

## How a host MVC project consumes these packs

1. The `.csproj` (e.g. `MyProject.Web`) carries a `package.json` referencing `@abp/aspnetcore.mvc.ui.theme.shared` (or theme-basic).
2. A `gulpfile.js` does:

```js gulpfile.js (host project, typical)
const gulp = require('gulp');
const path = require('path');
const cliResources = require("@abp/aspnetcore.mvc.ui/gulp/copy-resources");

gulp.task('default', () => cliResources(path.resolve(__dirname)));
```

3. `yarn install && yarn gulp` runs at build time; `wwwroot/libs/` is populated from `node_modules` according to the merged `mappings`.

`update-gulp.js` at `npm/` is the master script that walks every `.csproj`+`gulpfile.js` pair under the repository, updates the `@abp/*` versions with `npm-check-updates`, and re-runs `yarn gulp`. See [`/npm/abp-tooling`](/npm/abp-tooling).

## Versioning & publishing

Every pack ships at the same version as the rest of the ABP release — the value of `<Version>` inside the root `common.props`:

```xml common.props (excerpt)
<Project>
  <PropertyGroup>
    <Version>8.0.2</Version>
```

`publish-mvc.ps1` reads that value through `publish-utils.js`, then runs `npm run lerna -- version $Version --force-publish` which bumps every `packs/*/package.json` and every internal `@abp/*` range to the new value. `replace-with-tilde.js` then converts `^` to `~`, and `scripts/validate-versions.ts` fails the build if any pack disagrees:

```ts npm/scripts/validate-versions.ts (excerpt)
if (
  !excludedPackages.includes(pkgJson.name) &&
  pkgJson.version !== compareVersion
) {
  throwError(pkgJsonPath, pkgJson.name, pkgJson.version);
}
```

Finally `lerna exec npm publish` pushes every package to the registry in dependency order.
