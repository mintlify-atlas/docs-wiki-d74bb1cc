The `templates/app/angular` folder is the source for ABP's Angular single-page-application shell. It is **not** a standalone CLI template — `abp new -t app -u angular` materializes the layered ASP.NET Core solution under `templates/app/aspnet-core` *and* this Angular project side by side. The Angular workspace is a stock Angular 17 application generated by `@angular/cli` and then augmented with `@abp/ng.*` packages: `CoreModule.forRoot` consumes the typed `Environment` from `environments/environment.ts`, `AbpOAuthModule` plugs the OIDC authorization-code-with-PKCE flow into the HTTP client, `ThemeLeptonXModule` + `SideMenuLayoutModule` provide the chrome, and `AppRoutingModule` lazy-loads the ABP application modules (account, identity, tenant-management, setting-management) so the bundle stays small. The same `MyCompanyName.MyProjectName` placeholders the ASP.NET Core templates use are replaced by `SolutionRenameStep` here too — see `environment.ts`, where `clientId: 'MyProjectName_App'` becomes `Acme_BookStore_App` after generation.

<Info>
This page documents the Angular project as it lives in the repo. For the AppModule/AppRoutingModule design at runtime, lifecycle hooks, and how to add your own lazy module, see the ABP Angular [Overview](/ng/overview).
</Info>

## AppModule — providers and root imports

`AppModule` is the assembly point for every ABP Angular package. The two patterns to notice: `*ConfigModule.forRoot()` is the eager registration that runs at startup (registers translations, providers, side-menu entries), while the matching feature module (`AccountModule`, `IdentityModule`, …) is **lazy** and only loaded when its route activates.

```typescript templates/app/angular/src/app/app.module.ts lines icon="cube"
import { AccountConfigModule } from '@abp/ng.account/config';
import { CoreModule } from '@abp/ng.core';
import { registerLocale } from '@abp/ng.core/locale';
import { IdentityConfigModule } from '@abp/ng.identity/config';
import { SettingManagementConfigModule } from '@abp/ng.setting-management/config';
import { TenantManagementConfigModule } from '@abp/ng.tenant-management/config';
import { ThemeLeptonXModule } from '@abp/ng.theme.lepton-x';
import { SideMenuLayoutModule } from '@abp/ng.theme.lepton-x/layouts';
import { ThemeSharedModule } from '@abp/ng.theme.shared';
import { NgModule } from '@angular/core';
import { BrowserModule } from '@angular/platform-browser';
import { BrowserAnimationsModule } from '@angular/platform-browser/animations';
import { environment } from '../environments/environment';
import { AppRoutingModule } from './app-routing.module';
import { AppComponent } from './app.component';
import { APP_ROUTE_PROVIDER } from './route.provider';
import { FeatureManagementModule } from '@abp/ng.feature-management';
import { AbpOAuthModule } from '@abp/ng.oauth';
import { AccountLayoutModule } from '@abp/ng.theme.lepton-x/account';

@NgModule({
  imports: [
    BrowserModule,
    BrowserAnimationsModule,
    AppRoutingModule,
    CoreModule.forRoot({
      environment,
      registerLocaleFn: registerLocale(),
    }),
    AbpOAuthModule.forRoot(),
    ThemeSharedModule.forRoot(),
    AccountLayoutModule.forRoot(),
    AccountConfigModule.forRoot(),
    IdentityConfigModule.forRoot(),
    TenantManagementConfigModule.forRoot(),
    SettingManagementConfigModule.forRoot(),
    ThemeLeptonXModule.forRoot(),
    SideMenuLayoutModule.forRoot(),
    FeatureManagementModule.forRoot(),
  ],
  declarations: [AppComponent],
  providers: [APP_ROUTE_PROVIDER],
  bootstrap: [AppComponent],
})
export class AppModule {}
```

<AccordionGroup>
  <Accordion title="Why CoreModule.forRoot needs the environment">
    `CoreModule.forRoot({ environment, registerLocaleFn })` injects the typed `Environment` (the same object you exported from `environment.ts`) into ABP's `EnvironmentService`, where the rest of the framework reads `oAuthConfig`, `apis.default.url`, and `apis.default.rootNamespace`. Forgetting this call means `AbpOAuthModule` cannot find an issuer URL and the HTTP client cannot resolve API endpoints.
  </Accordion>
  <Accordion title="Why both ThemeLeptonXModule and SideMenuLayoutModule">
    `ThemeLeptonXModule` registers theme styles, color palettes, and dynamic-layout components. `SideMenuLayoutModule.forRoot()` registers the **layout** itself with ABP's `LayoutService` so that `<abp-dynamic-layout>` (in `AppComponent`) can resolve it at runtime. Swap `SideMenuLayoutModule` for `TopMenuLayoutModule` to change the chrome without touching the rest of the app.
  </Accordion>
  <Accordion title="Why AccountLayoutModule.forRoot is separate">
    The account pages (login, register, forgot password) use a **different** layout — full-screen, no side menu. `AccountLayoutModule.forRoot()` registers that layout under the `eLayoutType.account` key, which the lazy-loaded `@abp/ng.account` package activates for routes with `layout: 'account'`.
  </Accordion>
  <Accordion title="What APP_ROUTE_PROVIDER does">
    It is an `APP_INITIALIZER` factory that calls `routesService.add(...)` to register a single Home entry. Add more entries here to populate the side menu with your own pages (lazy modules contribute their own entries through their `*ConfigModule.forRoot()` registrations).
  </Accordion>
</AccordionGroup>

## AppRoutingModule — lazy module catalog

Every top-level route except Home is lazy-loaded from a published `@abp/ng.*` package. The `.forLazy()` static factory exists on each feature module to give it a chance to set up child providers without re-running the eager registrations from its companion `*ConfigModule`.

```typescript templates/app/angular/src/app/app-routing.module.ts lines icon="route"
import { NgModule } from '@angular/core';
import { RouterModule, Routes } from '@angular/router';

const routes: Routes = [
  {
    path: '',
    pathMatch: 'full',
    loadChildren: () => import('./home/home.module').then(m => m.HomeModule),
  },
  {
    path: 'account',
    loadChildren: () => import('@abp/ng.account').then(m => m.AccountModule.forLazy()),
  },
  {
    path: 'identity',
    loadChildren: () => import('@abp/ng.identity').then(m => m.IdentityModule.forLazy()),
  },
  {
    path: 'tenant-management',
    loadChildren: () =>
      import('@abp/ng.tenant-management').then(m => m.TenantManagementModule.forLazy()),
  },
  {
    path: 'setting-management',
    loadChildren: () =>
      import('@abp/ng.setting-management').then(m => m.SettingManagementModule.forLazy()),
  },
];

@NgModule({
  imports: [RouterModule.forRoot(routes, {})],
  exports: [RouterModule],
})
export class AppRoutingModule {}
```

| Route | Lazy module | What it provides |
| --- | --- | --- |
| `''` | `./home/home.module` | The local Home page (placeholder content). |
| `/account` | `@abp/ng.account` | Login, register, forgot/reset password, manage profile UI. |
| `/identity` | `@abp/ng.identity` | Users, roles, claim types, organization units management. |
| `/tenant-management` | `@abp/ng.tenant-management` | Tenants and connection-strings UI (host-side only). |
| `/setting-management` | `@abp/ng.setting-management` | Settings UI driven by `ISettingDefinitionProvider`. |

<Note>
The `feature-management` module is imported eagerly in `AppModule` (`FeatureManagementModule.forRoot()`) rather than being lazy-routed, because it exposes a **modal** that other modules open in-place rather than a routed page.
</Note>

## environment.ts — OIDC and API wiring

`environment.ts` is the **only** file you need to edit when pointing the SPA at a different ABP backend. The shape is enforced by the imported `Environment` type from `@abp/ng.core`, so missing fields are caught at compile time.

```typescript templates/app/angular/src/environments/environment.ts lines icon="gear"
import { Environment } from '@abp/ng.core';

const baseUrl = 'http://localhost:4200';

export const environment = {
  production: false,
  application: {
    baseUrl,
    name: 'MyProjectName',
    logoUrl: '',
  },
  oAuthConfig: {
    issuer: 'https://localhost:44305/',
    redirectUri: baseUrl,
    clientId: 'MyProjectName_App',
    responseType: 'code',
    scope: 'offline_access MyProjectName',
    requireHttps: true,
  },
  apis: {
    default: {
      url: 'https://localhost:44305',
      rootNamespace: 'MyCompanyName.MyProjectName',
    },
  },
} as Environment;
```

Two things worth highlighting:

- **`responseType: 'code'`** — authorization code with PKCE, the OIDC flow `AbpOAuthModule` expects. The matching backend (the `AuthServer` host in the layered template) has to have a client registration with the same `MyProjectName_App` client id, redirect URI `http://localhost:4200`, and `offline_access MyProjectName` scope.
- **The placeholder port `44305`** — `TemplateRandomSslPortStep` does not directly touch `environment.ts`, but `AngularEnvironmentFilePortChangeForSeparatedAuthServersStep` (under `Templates/App/`) rewrites the issuer URL to the randomized auth-server port when the layered solution is generated with `--separate-auth-server`.

`environment.prod.ts` is byte-identical except `production: true`.

```typescript templates/app/angular/src/app/app.component.ts lines icon="window"
import { Component } from '@angular/core';

@Component({
  selector: 'app-root',
  template: `
    <abp-loader-bar></abp-loader-bar>
    <abp-dynamic-layout></abp-dynamic-layout>
```

`AppComponent` is intentionally tiny. `<abp-dynamic-layout>` looks at the current activated route's `data.layout` (or falls back to the `application` layout) and renders the matching layout component from `SideMenuLayoutModule` / `AccountLayoutModule`. `<abp-loader-bar>` is the top-of-page progress indicator the HTTP interceptors drive.

## route.provider.ts — extending the side menu

```typescript templates/app/angular/src/app/route.provider.ts lines icon="bars"
import { RoutesService, eLayoutType } from '@abp/ng.core';
import { APP_INITIALIZER } from '@angular/core';

export const APP_ROUTE_PROVIDER = [
  { provide: APP_INITIALIZER, useFactory: configureRoutes, deps: [RoutesService], multi: true },
];

function configureRoutes(routesService: RoutesService) {
  return () => {
    routesService.add([
      {
        path: '/',
        name: '::Menu:Home',
        iconClass: 'fas fa-home',
        order: 1,
        layout: eLayoutType.application,
      },
    ]);
  };
}
```

`name: '::Menu:Home'` is a localization key — the empty resource prefix `::` resolves against the current ABP localization resource. The lazy account / identity / tenant-management / setting-management modules each add their own entries the same way through their config modules.

## SharedModule

The shared module is a re-exporter. It does not declare any components — its only job is to spare every feature module from re-importing the same five things.

```typescript templates/app/angular/src/app/shared/shared.module.ts lines icon="share-nodes"
import { CoreModule } from '@abp/ng.core';
import { NgbDropdownModule } from '@ng-bootstrap/ng-bootstrap';
import { NgModule } from '@angular/core';
import { ThemeSharedModule } from '@abp/ng.theme.shared';
import { NgxValidateCoreModule } from '@ngx-validate/core';

@NgModule({
  declarations: [],
  imports: [
    CoreModule,
    ThemeSharedModule,
    NgbDropdownModule,
    NgxValidateCoreModule
  ],
  exports: [
    CoreModule,
    ThemeSharedModule,
    NgbDropdownModule,
    NgxValidateCoreModule
  ],
  providers: []
})
export class SharedModule {}
```

`NgxValidateCoreModule` powers the reactive-forms validation directives ABP UI uses (`<val-errors>`). `NgbDropdownModule` is included because the LeptonX side menu uses Bootstrap-style dropdowns under the hood.

## How the CLI generates the Angular app

The Angular project lives **inside** the layered `app` template archive. When the CLI runs `abp new -t app -u angular`, [`AppTemplateBase.GetCustomSteps`](/templates/template-structure-and-replacements) sees `UiFramework.Angular` and:

<Steps>
  <Step title="Removes MVC and Blazor projects">
    `RemoveProjectFromSolutionStep` deletes `MyCompanyName.MyProjectName.Web`, `.Blazor`, `.Blazor.Server`, `.Blazor.WebAssembly` (and their MAUI Blazor counterparts) from the solution.
  </Step>
  <Step title="Adds the angular symbol">
    `context.Symbols.Add("ui:angular")` is added so `TemplateCodeDeleteStep` can keep or strip `// <TEMPLATE-REMOVE-IF-NOT-UI-ANGULAR>` blocks in `appsettings.json` and module classes.
  </Step>
  <Step title="MakeProxyJsonFileEmbeddedStep skipped">
    The Angular generation does not embed JSON proxies — generation is driven by the separate `abp generate-proxy -t ng` command at development time.
  </Step>
  <Step title="AngularEnvironmentFilePortChangeForSeparatedAuthServersStep">
    Only runs if `--separate-auth-server` was passed. Rewrites the `issuer` URL in `environment.ts` and `environment.prod.ts` to the randomized auth-server port that `TemplateRandomSslPortStep` produced.
  </Step>
  <Step title="SolutionRenameStep">
    Replaces `MyCompanyName.MyProjectName` in `environment.ts.rootNamespace`, `MyProjectName` in `clientId` / `scope` / `application.name`, and the workspace project name in `angular.json` and `package.json`.
  </Step>
</Steps>

## Customizing the template

The Angular project is pure files — anything you add to `templates/app/angular/src/` ships in the generated SPA after the template archive is rebuilt. Common edits:

- Add a feature module: create `src/app/<name>/<name>.module.ts` with its own routing module, then register a lazy route in `app-routing.module.ts`.
- Add a side-menu entry: extend `route.provider.ts` with another `routesService.add(...)` call.
- Switch layouts: replace `SideMenuLayoutModule.forRoot()` with `TopMenuLayoutModule.forRoot()` from `@abp/ng.theme.lepton-x/layouts`.
- Switch themes: replace `ThemeLeptonXModule` with `ThemeBasicModule` from `@abp/ng.theme.basic` (and adjust the `angular.json` stylesheet bundles accordingly).
