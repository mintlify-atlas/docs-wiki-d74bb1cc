The Angular workspace under `modules/cms-kit/angular/` is the open‚Äësource Angular SPA companion to the CMS Kit module. It is structured the same way as every other ABP Angular library: a `cms-kit` *library* project ships two NPM packages ‚Äî the runtime module and a forRoot config ‚Äî and a sibling `dev-app` application project hosts them so a contributor can `ng serve` and see the result. The library itself is intentionally tiny in this repo ‚Äî it provides a scaffold (`CmsKitModule` plus a stub `CmsKitComponent`), a `CmsKitConfigModule.forRoot()` that registers a `/cms-kit` route under the application layout, and a `CmsKitService` that wraps `RestService`. Most of the public‚Äësite logic for blogs, comments, reactions, and so on is rendered server‚Äëside by the Razor public web project; the Angular pieces here are the SPA hooks an integrator extends. This page maps every project under `projects/`, lists the registration entry points, and shows how the dev harness consumes the library.

<Info>
Source root: [`modules/cms-kit/angular/`](https://github.com/abpframework/abp/tree/dev/modules/cms-kit/angular). The full Angular UI for the commercial CMS Kit Pro module ships separately under the `abp.io` commercial distribution; the open‚Äësource angular project here is the scaffolding.
</Info>

- **`dev-app`** ‚Äî `projectType: application`, `ng serve` target, depends on `cms-kit` and on the standard ABP modules (`@abp/ng.core`, `@abp/ng.theme.basic`, `@abp/ng.account`, `@abp/ng.identity`, `@abp/ng.setting-management`, `@abp/ng.tenant-management`).

## The runtime module

The library's main entry point exports a thin `NgModule`:

```ts title="modules/cms-kit/angular/projects/cms-kit/src/public-api.ts"
/*
 * Public API Surface of cms-kit
 */

export * from './lib/components/cms-kit.component';
export * from './lib/services/cms-kit.service';
export * from './lib/cms-kit.module';
```

```ts title="modules/cms-kit/angular/projects/cms-kit/src/lib/cms-kit.module.ts"
import { NgModule, NgModuleFactory, ModuleWithProviders } from '@angular/core';
import { CoreModule, LazyModuleFactory } from '@abp/ng.core';
import { ThemeSharedModule } from '@abp/ng.theme.shared';
import { CmsKitComponent } from './components/cms-kit.component';
import { CmsKitRoutingModule } from './cms-kit-routing.module';

@NgModule({
  declarations: [CmsKitComponent],
  imports: [CoreModule, ThemeSharedModule, CmsKitRoutingModule],
  exports: [CmsKitComponent],
})
export class CmsKitModule {
  static forChild(): ModuleWithProviders<CmsKitModule> {
    return {
      ngModule: CmsKitModule,
      providers: [],
    };
  }

  static forLazy(): NgModuleFactory<CmsKitModule> {
    return new LazyModuleFactory(CmsKitModule.forChild());
  }
}
```

Three things worth noticing:

1. **The module is designed to be lazy‚Äëloaded.** `forLazy()` returns a `LazyModuleFactory` so a parent application can mount it via `loadChildren: () => CmsKitModule.forLazy()`, which is how `@abp/ng.core` expects feature modules to be wired.
2. **It imports `CoreModule` and `ThemeSharedModule`.** The first gives access to `RestService`, current‚Äëuser, permission directives, and so on; the second contains the shared theme primitives used by every ABP page.
3. **`forChild()` exists for completeness but currently provides nothing.** When you extend `CmsKitModule` in your own project to add real components, you would add per‚Äëmount providers here.

The routing module is just a single route that renders `CmsKitComponent` inside the standard ABP layout:

```ts title="modules/cms-kit/angular/projects/cms-kit/src/lib/cms-kit-routing.module.ts"
import { NgModule } from '@angular/core';
import { DynamicLayoutComponent } from '@abp/ng.core';
import { Routes, RouterModule } from '@angular/router';
import { CmsKitComponent } from './components/cms-kit.component';

const routes: Routes = [
  {
    path: '',
    pathMatch: 'full',
    component: DynamicLayoutComponent,
    children: [
      {
        path: '',
        component: CmsKitComponent,
      },
    ],
  },
];

@NgModule({
  imports: [RouterModule.forChild(routes)],
  exports: [RouterModule],
})
export class CmsKitRoutingModule {}
```

`DynamicLayoutComponent` is `@abp/ng.core`'s wrapper that picks the right layout (application / account / empty) from the current `eLayoutType`. The route entry from the config module (below) declared `layout: eLayoutType.application`, so the component renders inside the dashboard chrome.

`CmsKitComponent` is the scaffold itself ‚Äî it pings `/api/CmsKit/sample` to prove the wiring works:

```ts title="modules/cms-kit/angular/projects/cms-kit/src/lib/components/cms-kit.component.ts"
import { Component, OnInit } from '@angular/core';
import { CmsKitService } from '../services/cms-kit.service';

@Component({
  selector: 'lib-cms-kit',
  template: ` <p>cms-kit works!</p> `,
  styles: [],
})
export class CmsKitComponent implements OnInit {
  constructor(private service: CmsKitService) {}

  ngOnInit(): void {
    this.service.sample().subscribe(console.log);
  }
}
```

```ts title="modules/cms-kit/angular/projects/cms-kit/src/lib/services/cms-kit.service.ts"
import { Injectable } from '@angular/core';
import { RestService } from '@abp/ng.core';

@Injectable({
  providedIn: 'root',
})
export class CmsKitService {
  apiName = 'CmsKit';

  constructor(private restService: RestService) {}

  sample() {
    return this.restService.request<void, any>(
      { method: 'GET', url: '/api/CmsKit/sample' },
      { apiName: this.apiName }
    );
  }
}
```

The pattern ‚Äî `restService.request({method, url}, {apiName})` ‚Äî is the same one ABP's Angular generator emits for every typed proxy. The `apiName` matches the `RemoteServiceName` constant on the C# side (`CmsKitCommon`, `CmsKitAdmin`, or `CmsKitPublic` for each tier ‚Äî see [Admin / Public split](/modules/cms-kit/admin-public-split)) so HTTP base URLs are resolved per remote service.

## The config module (forRoot)

The secondary entry point ‚Äî `@my-company-name/cms-kit/config` ‚Äî registers a sidebar entry through `RoutesService`:

```ts title="modules/cms-kit/angular/projects/cms-kit/config/src/cms-kit-config.module.ts"
import { ModuleWithProviders, NgModule } from '@angular/core';
import { MY_PROJECT_NAME_ROUTE_PROVIDERS } from './providers/route.provider';

@NgModule()
export class CmsKitConfigModule {
  static forRoot(): ModuleWithProviders<CmsKitConfigModule> {
    return {
      ngModule: CmsKitConfigModule,
      providers: [MY_PROJECT_NAME_ROUTE_PROVIDERS],
    };
  }
}
```

```ts title="modules/cms-kit/angular/projects/cms-kit/config/src/providers/route.provider.ts"
import { eLayoutType, RoutesService } from '@abp/ng.core';
import { APP_INITIALIZER } from '@angular/core';
import { eCmsKitRouteNames } from '../enums/route-names';

export const MY_PROJECT_NAME_ROUTE_PROVIDERS = [
    {
        provide: APP_INITIALIZER,
        useFactory: configureRoutes,
        deps: [RoutesService],
        multi: true,
    },
];

export function configureRoutes(routesService: RoutesService) {
    return () => {
        routesService.add([
            {
                path: '/cms-kit',
                name: eCmsKitRouteNames.CmsKit,
                iconClass: 'fas fa-book',
                layout: eLayoutType.application,
                order: 3,
            },
        ]);
    };
}
```

```ts title="modules/cms-kit/angular/projects/cms-kit/config/src/enums/route-names.ts"
export const enum eCmsKitRouteNames {
  CmsKit = 'CmsKit',
}
```

Two patterns to observe:

1. **`APP_INITIALIZER` runs on app boot**, registering the `/cms-kit` route metadata (path, name, icon, layout, order) with the ABP router. This is *separate* from Angular's own `Routes` ‚Äî it tells the navigation menu what to show; the actual route handler is wired by `app-routing.module.ts` below.
2. **The `MY_PROJECT_NAME_ROUTE_PROVIDERS` constant name** is a leftover from the project template generator ‚Äî it is still exported under that name from `providers/index.ts` so the dev‚Äëapp can pull it in. Integrators usually rename it.

## The dev-app harness

The sibling `dev-app` application wires up the library so contributors can run `ng serve` and exercise it:

```ts title="modules/cms-kit/angular/projects/dev-app/src/app/app.module.ts"
import { AccountConfigModule }            from '@abp/ng.account/config';
import { CoreModule }                     from '@abp/ng.core';
import { IdentityConfigModule }           from '@abp/ng.identity/config';
import { SettingManagementConfigModule }  from '@abp/ng.setting-management/config';
import { TenantManagementConfigModule }   from '@abp/ng.tenant-management/config';
import { ThemeBasicModule }               from '@abp/ng.theme.basic';
import { ThemeSharedModule }              from '@abp/ng.theme.shared';
import { NgModule }                       from '@angular/core';
import { BrowserModule }                  from '@angular/platform-browser';
import { BrowserAnimationsModule }        from '@angular/platform-browser/animations';
import { CmsKitConfigModule }             from '@my-company-name/cms-kit/config';
import { environment }                    from '../environments/environment';
import { AppRoutingModule }               from './app-routing.module';
import { AppComponent }                   from './app.component';
import { APP_ROUTE_PROVIDER }             from './route.provider';

@NgModule({
  imports: [
    BrowserModule,
    BrowserAnimationsModule,
    AppRoutingModule,
    CoreModule.forRoot({
      environment,
      sendNullsAsQueryParam: false,
      skipGetAppConfiguration: false,
    }),
    ThemeSharedModule.forRoot(),
    AccountConfigModule.forRoot(),
    IdentityConfigModule.forRoot(),
    CmsKitConfigModule.forRoot(),
    TenantManagementConfigModule.forRoot(),
    SettingManagementConfigModule.forRoot(),
    ThemeBasicModule.forRoot(),
  ],
  providers: [APP_ROUTE_PROVIDER],
  declarations: [AppComponent],
  bootstrap: [AppComponent],
})
export class AppModule {}
```

Notice the import order matches the convention every ABP Angular app follows:

1. `CoreModule.forRoot({...})` first ‚Äî the `environment` block carries the API base URLs and the OIDC client config.
2. `ThemeSharedModule.forRoot()` ‚Äî shared theme tokens.
3. The standard config modules ‚Äî Account, Identity, Tenant Management, Setting Management ‚Äî and the project's own `CmsKitConfigModule.forRoot()`.
4. `ThemeBasicModule.forRoot()` last ‚Äî it picks up the layout components.

The routing module mounts `CmsKitModule.forLazy()` under `/cms-kit`:

```ts title="modules/cms-kit/angular/projects/dev-app/src/app/app-routing.module.ts"
const routes: Routes = [
  {
    path: '',
    pathMatch: 'full',
    loadChildren: () => import('./home/home.module').then((m) => m.HomeModule),
  },
  // account, identity, tenant-management, setting-management routes ...
  {
    path: 'cms-kit',
    loadChildren: () =>
      import('@my-company-name/cms-kit').then((m) =>
        m.CmsKitModule.forLazy()
      ),
  },
];
```

`m.CmsKitModule.forLazy()` returns the `LazyModuleFactory` declared in `cms-kit.module.ts` ‚Äî Angular treats it like a normal lazy module from the router's point of view.

## Building and serving

The library is built with ng‚Äëpackagr:

```bash
# from modules/cms-kit/angular
yarn install
ng build cms-kit          # writes dist/cms-kit and dist/cms-kit/config
ng serve dev-app          # serves the dev harness at http://localhost:4200
```

The two‚Äëartifact output is consumed by integrators with a single dependency on `@my-company-name/cms-kit` plus a deep import for the secondary entry point:

```ts
import { CmsKitModule }       from '@my-company-name/cms-kit';
import { CmsKitConfigModule } from '@my-company-name/cms-kit/config';
```

The package name in this repo (`@my-company-name/cms-kit`) is template‚Äëgenerated; commercial Volo distributions ship under `@volo/cms-kit` (and similar).

## Extending the Angular UI

Because this open‚Äësource workspace ships only a scaffold, integrators typically:

1. **Generate typed proxies** with `abp generate-proxy -t ng` against the running back‚Äëend. This emits `models/`, `services/`, and `proxy/` folders under `cms-kit/src/lib/` mirroring `BlogAdminAppService`, `BlogPostAdminAppService`, `CommentPublicAppService`, etc.
2. **Replace `CmsKitComponent`** with a routed page tree (`/cms-kit/blogs`, `/cms-kit/pages`, ‚Ä¶) that calls into the generated proxies.
3. **Extend `CmsKitConfigModule.forRoot()`** to register a sidebar entry per page, each guarded by the matching `CmsKitAdminPermissions.*` permission. The `RoutesService` accepts a `requiredPolicy` field for permission‚Äëbased navigation.

The commercial Lepton Theme / Lepton X CMS Kit Pro UI follows exactly this pattern ‚Äî what you see in this open‚Äësource workspace is the minimum set of pieces an integrator needs to bootstrap their own.
