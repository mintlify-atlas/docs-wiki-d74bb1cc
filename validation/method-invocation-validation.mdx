The `IValidationEnabled` marker, together with the `ValidationInterceptor` and
`MethodInvocationValidator`, is how ABP automatically validates the arguments of
every public method on application services, controllers and any class that
opts in. The interceptor sits in ABP's dynamic-proxy chain (see
[Castle DynamicProxy integration](/di/castle-dynamic-proxy)), so validation runs
*before* the method body executes — and a single failure surfaces as a single
`AbpValidationException` with the full list of errors attached. This page traces
the registration hook, the interceptor, the per-parameter loop and the
disable / enable rules using the actual source under
`framework/src/Volo.Abp.Validation/Volo/Abp/Validation/`.

[validation overview](/validation/overview).

## Opting in: `IValidationEnabled`

A service is intercepted if (and only if) it implements `IValidationEnabled`:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/IValidationEnabled.cs
namespace Volo.Abp.Validation;

public interface IValidationEnabled
{

}
```

This marker is intentionally empty — the type's *presence* is the configuration.
Application service base classes (e.g. `ApplicationService`) implement it
transitively, which is why every application service in an ABP solution gets
validation automatically.

## Interceptor registration

The module installs an `OnRegistered` callback that runs once for each service
registration. The registrar inspects the implementation type and, if it derives
from `IValidationEnabled` and is not in `DynamicProxyIgnoreTypes`, adds
`ValidationInterceptor` to that service's interceptor list:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/ValidationInterceptorRegistrar.cs
public static class ValidationInterceptorRegistrar
{
    public static void RegisterIfNeeded(IOnServiceRegistredContext context)
    {
        if (ShouldIntercept(context.ImplementationType))
        {
            context.Interceptors.TryAdd<ValidationInterceptor>();
        }
    }

    private static bool ShouldIntercept(Type type)
    {
        return !DynamicProxyIgnoreTypes.Contains(type) && typeof(IValidationEnabled).IsAssignableFrom(type);
    }
}
```

The hook itself is wired in the module:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/AbpValidationModule.cs
public override void PreConfigureServices(ServiceConfigurationContext context)
{
    context.Services.OnRegistered(ValidationInterceptorRegistrar.RegisterIfNeeded);
    AutoAddObjectValidationContributors(context.Services);
}
```

<Note>
`context.Interceptors.TryAdd<TInterceptor>()` records the interceptor in ABP's
DI metadata; the actual proxy is generated by the container — Autofac through
the [Castle DynamicProxy adapter](/di/castle-dynamic-proxy) — when the service
is resolved. This means `ValidationInterceptor` can be a normal
`ITransientDependency` without any container-specific binding.
</Note>

## The interceptor

`ValidationInterceptor` is a minimal `AbpInterceptor`. It awaits the validator,
then proceeds with the call. If the validator throws, the inner method body is
never executed.

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/ValidationInterceptor.cs
public class ValidationInterceptor : AbpInterceptor, ITransientDependency
{
    private readonly IMethodInvocationValidator _methodInvocationValidator;

    public ValidationInterceptor(IMethodInvocationValidator methodInvocationValidator)
    {
        _methodInvocationValidator = methodInvocationValidator;
    }

    public override async Task InterceptAsync(IAbpMethodInvocation invocation)
    {
        await ValidateAsync(invocation);
        await invocation.ProceedAsync();
    }

    protected virtual async Task ValidateAsync(IAbpMethodInvocation invocation)
    {
        await _methodInvocationValidator.ValidateAsync(
            new MethodInvocationValidationContext(
                invocation.TargetObject,
                invocation.Method,
                invocation.Arguments
            )
        );
    }
}
```

The `IAbpMethodInvocation` abstraction is the same one used by every
ABP interceptor — auditing, unit of work, authorization. See
[Castle DynamicProxy integration](/di/castle-dynamic-proxy) for how
`invocation.TargetObject`, `invocation.Method` and `invocation.Arguments` are
populated.

## The per-invocation context

`MethodInvocationValidationContext` extends `AbpValidationResult` so the
`Errors` list is part of the context — different participants can share the
same list during a single call.

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/MethodInvocationValidationContext.cs
public class MethodInvocationValidationContext : AbpValidationResult
{
    public object TargetObject { get; }

    public MethodInfo Method { get; }

    public object[] ParameterValues { get; }

    public ParameterInfo[] Parameters { get; }

    public MethodInvocationValidationContext(object targetObject, MethodInfo method, object[] parameterValues)
    {
        TargetObject = targetObject;
        Method = method;
        ParameterValues = parameterValues;
        Parameters = method.GetParameters();
    }
}
```

The base type:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/AbpValidationResult.cs
public class AbpValidationResult : IAbpValidationResult
{
    public List<ValidationResult> Errors { get; }

    public AbpValidationResult()
    {
        Errors = new List<ValidationResult>();
    }
}
```

## `IMethodInvocationValidator`

The validator is a one-method abstraction so projects can replace it:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/IMethodInvocationValidator.cs
public interface IMethodInvocationValidator
{
    Task ValidateAsync(MethodInvocationValidationContext context);
}
```

The default implementation is `MethodInvocationValidator` — itself transient.

## The validation algorithm

`MethodInvocationValidator.ValidateAsync` runs a short sequence of guard checks,
then walks each parameter through `IObjectValidator.GetErrorsAsync`. The whole
method:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/MethodInvocationValidator.cs
public virtual async Task ValidateAsync(MethodInvocationValidationContext context)
{
    Check.NotNull(context, nameof(context));

    if (context.Parameters.IsNullOrEmpty())
    {
        return;
    }

    if (!context.Method.IsPublic)
    {
        return;
    }

    if (IsValidationDisabled(context))
    {
        return;
    }

    if (context.Parameters.Length != context.ParameterValues.Length)
    {
        throw new Exception("Method parameter count does not match with argument count!");
    }

    //todo: consider to remove this condition
    if (context.Errors.Any() && HasSingleNullArgument(context))
    {
        ThrowValidationError(context);
    }

    await AddMethodParameterValidationErrorsAsync(context);

    if (context.Errors.Any())
    {
        ThrowValidationError(context);
    }
}
```

The guard rules, in order:

| Step | What is checked | Effect when true |
| --- | --- | --- |
| 1 | `context.Parameters` empty | Skip — nothing to validate. |
| 2 | `context.Method.IsPublic == false` | Skip — non-public methods aren't validated even on `IValidationEnabled` types. |
| 3 | `IsValidationDisabled(context)` | Skip when `[DisableValidation]` applies (and `[EnableValidation]` is absent). |
| 4 | Parameter/value length mismatch | Throw `Exception("Method parameter count does not match with argument count!")` — a sanity check. |
| 5 | Single null argument and errors already present | Throw immediately. |
| 6 | Per-parameter validation collects errors | Throw `AbpValidationException` if any error was added. |

### Disable / enable

`IsValidationDisabled` walks the attributes via ABP's reflection helper. The
`[EnableValidation]` attribute on a method wins over a `[DisableValidation]` on
the declaring class:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/MethodInvocationValidator.cs
protected virtual bool IsValidationDisabled(MethodInvocationValidationContext context)
{
    if (context.Method.IsDefined(typeof(EnableValidationAttribute), true))
    {
        return false;
    }

    if (ReflectionHelper.GetSingleAttributeOfMemberOrDeclaringTypeOrDefault<DisableValidationAttribute>(context.Method) != null)
    {
        return true;
    }

    return false;
}
```

| Class-level attribute | Method-level attribute | Result |
| --- | --- | --- |
| *(none)* | *(none)* | Validate |
| `[DisableValidation]` | *(none)* | **Skip** |
| `[DisableValidation]` | `[EnableValidation]` | Validate |
| `[DisableValidation]` | `[DisableValidation]` | **Skip** |
| *(none)* | `[DisableValidation]` | **Skip** |

### Per-parameter loop

For every parameter, the validator asks `IObjectValidator` for the errors and
appends them to the shared list. Nulls are allowed when the parameter is
optional, an `out` parameter, or a primitive (including enums):

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/MethodInvocationValidator.cs
protected virtual async Task AddMethodParameterValidationErrorsAsync(MethodInvocationValidationContext context)
{
    for (var i = 0; i < context.Parameters.Length; i++)
    {
        await AddMethodParameterValidationErrorsAsync(context, context.Parameters[i], context.ParameterValues[i]);
    }
}

protected virtual async Task AddMethodParameterValidationErrorsAsync(IAbpValidationResult context, ParameterInfo parameterInfo, object parameterValue)
{
    var allowNulls = parameterInfo.IsOptional ||
                     parameterInfo.IsOut ||
                     TypeHelper.IsPrimitiveExtended(parameterInfo.ParameterType, includeEnums: true);

    context.Errors.AddRange(
         await _objectValidator.GetErrorsAsync(
            parameterValue,
            parameterInfo.Name,
            allowNulls
        )
    );
}
```

Each call goes through `ObjectValidator.GetErrorsAsync`, which iterates the
registered `IObjectValidationContributor`s — `DataAnnotationObjectValidationContributor`
by default, plus `FluentObjectValidationContributor` when
[Volo.Abp.FluentValidation](/validation/fluent-validation-integration) is
referenced.

### Throwing

`ThrowValidationError` is one line:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/MethodInvocationValidator.cs
protected virtual void ThrowValidationError(MethodInvocationValidationContext context)
{
    throw new AbpValidationException(
        "Method arguments are not valid! See ValidationErrors for details.",
        context.Errors
    );
}
```

Because the exception type implements `IExceptionWithSelfLogging` (see
[the overview page](/validation/overview)), all `ValidationResult.ErrorMessage`
values are written to the log in a single entry.

## Flow diagram

```mermaid
sequenceDiagram
    participant Caller
    participant Proxy as Castle proxy
    participant Interceptor as ValidationInterceptor
    participant MIV as MethodInvocationValidator
    participant OV as IObjectValidator
    participant Contrib as IObjectValidationContributor(s)
    participant Target as Real method

    Caller->>Proxy: foo.CreateAsync(dto)
    Proxy->>Interceptor: InterceptAsync(invocation)
    Interceptor->>MIV: ValidateAsync(context)
    MIV->>MIV: guard: Parameters empty? IsPublic? IsValidationDisabled?
    loop For each parameter
        MIV->>OV: GetErrorsAsync(value, name, allowNull)
        OV->>Contrib: AddErrorsAsync(ObjectValidationContext)
        Contrib-->>OV: context.Errors populated
        OV-->>MIV: List&lt;ValidationResult&gt;
    end
    alt context.Errors.Any()
        MIV-->>Interceptor: throw AbpValidationException
        Interceptor-->>Proxy: exception propagates
        Proxy-->>Caller: AbpValidationException
    else
        MIV-->>Interceptor: done
        Interceptor->>Proxy: invocation.ProceedAsync()
        Proxy->>Target: real call
        Target-->>Caller: result
    end
```

## `IValidatableObject` support

Because each parameter ultimately flows through
`DataAnnotationObjectValidationContributor.AddErrors`, any DTO that implements
`System.ComponentModel.DataAnnotations.IValidatableObject` participates without
extra plumbing:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/DataAnnotationObjectValidationContributor.cs
public void AddErrors(List<ValidationResult> errors, object validatingObject)
{
    var properties = TypeDescriptor.GetProperties(validatingObject).Cast<PropertyDescriptor>();

    foreach (var property in properties)
    {
        AddPropertyErrors(validatingObject, property, errors);
    }

    if (validatingObject is IValidatableObject validatableObject)
    {
        errors.AddRange(
            validatableObject.Validate(new ValidationContext(validatableObject, ServiceProvider, null))
        );
    }
}
```

The `ValidationContext` passed to `Validate` exposes the current
`IServiceProvider` (a scoped one — see `ObjectValidator.GetErrorsAsync`), so
`IValidatableObject` implementations can pull additional services:

```csharp
public class CreateOrderDto : IValidatableObject
{
    public Guid CustomerId { get; set; }

    public List<OrderLineDto> Lines { get; set; }

    public IEnumerable<ValidationResult> Validate(ValidationContext context)
    {
        if (Lines == null || Lines.Count == 0)
        {
            yield return new ValidationResult("Order must contain at least one line.", new[] { nameof(Lines) });
        }

        var customers = context.GetRequiredService<ICustomerLookupService>();
        if (!customers.Exists(CustomerId))
        {
            yield return new ValidationResult("Unknown customer.", new[] { nameof(CustomerId) });
        }
    }
}
```

## Recursive parameter validation

When a parameter is a graph (DTO with nested DTOs, collections), the
data-annotation contributor descends — capped at 8 levels — and skips primitives
and `IQueryable`:

```csharp framework/src/Volo.Abp.Validation/Volo/Abp/Validation/DataAnnotationObjectValidationContributor.cs
public const int MaxRecursiveParameterValidationDepth = 8;

protected virtual void ValidateObjectRecursively(List<ValidationResult> errors, object? validatingObject, int currentDepth)
{
    if (currentDepth > MaxRecursiveParameterValidationDepth)
    {
        return;
    }

    if (validatingObject == null)
    {
        return;
    }

    AddErrors(errors, validatingObject);

    //Validate items of enumerable
    if (validatingObject is IEnumerable enumerable)
    {
        if (!(enumerable is IQueryable))
        {
            foreach (var item in enumerable)
            {
                //Do not recursively validate for primitive objects
                if (item == null || TypeHelper.IsPrimitiveExtended(item.GetType()))
                {
                    break;
                }

                ValidateObjectRecursively(errors, item, currentDepth + 1);
            }
        }

        return;
    }

    var validatingObjectType = validatingObject.GetType();

    //Do not recursively validate for primitive objects
    if (TypeHelper.IsPrimitiveExtended(validatingObjectType))
    {
        return;
    }

    if (Options.IgnoredTypes.Any(t => t.IsInstanceOfType(validatingObject)))
    {
        return;
    }

    var properties = TypeDescriptor.GetProperties(validatingObject).Cast<PropertyDescriptor>();
    foreach (var property in properties)
    {
        if (property.Attributes.OfType<DisableValidationAttribute>().Any())
        {
            continue;
        }

        ValidateObjectRecursively(errors, property.GetValue(validatingObject), currentDepth + 1);
    }
}
```

<Tip>
Use `AbpValidationOptions.IgnoredTypes` to skip whole graphs — for example to
avoid traversing a tracked EF Core entity that lives on a DTO:

```csharp
Configure<AbpValidationOptions>(options =>
{
    options.IgnoredTypes.Add(typeof(IQueryable));
});
```
</Tip>

## A worked example

Combining everything: an application service with data annotations and an
`IValidatableObject` DTO.

```csharp
public class BookAppService : ApplicationService, IBookAppService // IBookAppService : IValidationEnabled
{
    public Task<BookDto> CreateAsync(CreateBookDto input)
    {
        // input has already been validated by the interceptor before this line
        return /* ... */;
    }
}

public class CreateBookDto : IValidatableObject
{
    [Required]
    [StringLength(128)]
    public string Name { get; set; }

    [Range(0.0, 1_000_000.0)]
    public decimal Price { get; set; }

    public IEnumerable<ValidationResult> Validate(ValidationContext context)
    {
        if (Name?.StartsWith("Untitled") == true)
        {
            yield return new ValidationResult(
                "Books must have a real name.",
                new[] { nameof(Name) }
            );
        }
    }
}
```

When a controller posts a `null` body, or a `Name` that is too long, the
interceptor produces a single `AbpValidationException` containing every error.
The web layer's [exception handling pipeline](/web/exception-handling) converts
it into HTTP 400 with a `RemoteServiceErrorResponse` that lists every offending
member name.

## Replacing the validator

`MethodInvocationValidator` is registered conventionally; subclass it and
replace the registration to change behavior — for example to add custom logging
of every failed validation:

```csharp
[ExposeServices(typeof(IMethodInvocationValidator))]
public class LoggingMethodInvocationValidator : MethodInvocationValidator, ITransientDependency
{
    private readonly ILogger<LoggingMethodInvocationValidator> _logger;

    public LoggingMethodInvocationValidator(IObjectValidator objectValidator, ILogger<LoggingMethodInvocationValidator> logger)
        : base(objectValidator)
    {
        _logger = logger;
    }

    protected override void ThrowValidationError(MethodInvocationValidationContext context)
    {
        _logger.LogWarning("Validation failed for {Method} with {Count} errors",
            context.Method.Name, context.Errors.Count);
        base.ThrowValidationError(context);
    }
}
```
