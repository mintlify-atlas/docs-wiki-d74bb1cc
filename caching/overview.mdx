ABP's caching layer is a thin, opinionated wrapper around `Microsoft.Extensions.Caching.Distributed.IDistributedCache`. It is delivered by the `Volo.Abp.Caching` package and gives application code a **typed**, **multi-tenant-aware**, **unit-of-work-aware** cache that serializes objects to bytes once, keys them in a single canonical way, and threads exception handling through the framework's [exception notifier](/utilities/exception-handling). On top of that, `Volo.Abp.Caching.StackExchangeRedis` swaps the in-memory backing store for Redis without changing a single line of application code — every call site still talks to `IDistributedCache<TCacheItem>`.

This page introduces the moving parts: the `AbpCachingModule` that wires everything into DI, the two generic `IDistributedCache<,>` interfaces that application services consume, the `CacheNameAttribute` and `IgnoreMultiTenancyAttribute` that decorate cache-item POCOs, and the `AbpDistributedCacheOptions` that lets a module register global defaults and per-cache configurators. Implementation details — the `DistributedCache<,>` class, the serializer, and the key normalizer — are covered in [Distributed cache](/caching/distributed-cache), and the Redis backing store is covered in [StackExchange.Redis](/caching/stackexchange-redis).

## Package layout

`Volo.Abp.Caching` is intentionally small — a single namespace, a single module, and a handful of abstractions. The whole module lives under `framework/src/Volo.Abp.Caching/Volo/Abp/Caching`.

| File | Type | Role |
| --- | --- | --- |
| `AbpCachingModule.cs` | `AbpModule` | Registers DI bindings, defaults, and depends on `AbpMultiTenancyModule` and `AbpJsonModule`. |
| `IDistributedCache.cs` | Interfaces | `IDistributedCache<TCacheItem>` and `IDistributedCache<TCacheItem, TCacheKey>` — the typed cache API. |
| `DistributedCache.cs` | Class | `DistributedCache<TCacheItem>` and `DistributedCache<TCacheItem, TCacheKey>` — default implementations. |
| `AbpDistributedCacheOptions.cs` | Options | `HideErrors`, `KeyPrefix`, `GlobalCacheEntryOptions`, `CacheConfigurators`. |
| `CacheNameAttribute.cs` | Attribute | Overrides the cache name derived from the cache item's CLR type. |
| `IDistributedCacheSerializer.cs` | Interface | Serializer abstraction. |
| `Utf8JsonDistributedCacheSerializer.cs` | Class | Default JSON-over-UTF-8 serializer. |
| `IDistributedCacheKeyNormalizer.cs` | Interface | Builds the final string key from `(key, cacheName, ignoreMultiTenancy)`. |
| `DistributedCacheKeyNormalizer.cs` | Class | Default normalizer; prefixes tenant id and `KeyPrefix`. |
| `DistributedCacheKeyNormalizeArgs.cs` | Class | Value object passed to the normalizer. |
| `ICacheSupportsMultipleItems.cs` | Interface | Opt-in batch API; implemented by `AbpRedisCache`. |
| `UnitOfWorkCacheItem.cs` | Class | Per-UoW snapshot of a value with an `IsRemoved` flag. |
| `UnitOfWorkCacheItemExtensions.cs` | Extensions | `GetUnRemovedValueOrNull()` helper used by `DistributedCache<,>`. |

The matching attribute used to opt-out of tenant-prefixed keys lives in the multi-tenancy abstractions:

| File | Type | Role |
| --- | --- | --- |
| `framework/src/Volo.Abp.MultiTenancy.Abstractions/Volo/Abp/MultiTenancy/IgnoreMultiTenancyAttribute.cs` | Attribute | Marker on a cache item type; suppresses tenant prefix in normalized keys. |

## `AbpCachingModule`

The module wires the standard ASP.NET Core memory and distributed-memory caches, registers the open generics for the ABP typed cache, and seeds a default 20-minute sliding expiration into `AbpDistributedCacheOptions.GlobalCacheEntryOptions`.

```csharp title="framework/src/Volo.Abp.Caching/Volo/Abp/Caching/AbpCachingModule.cs"
[DependsOn(
    typeof(AbpThreadingModule),
    typeof(AbpSerializationModule),
    typeof(AbpUnitOfWorkModule),
    typeof(AbpMultiTenancyModule),
    typeof(AbpJsonModule))]
public class AbpCachingModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        context.Services.AddMemoryCache();
        context.Services.AddDistributedMemoryCache();

        context.Services.AddSingleton(typeof(IDistributedCache<>), typeof(DistributedCache<>));
        context.Services.AddSingleton(typeof(IDistributedCache<,>), typeof(DistributedCache<,>));

        context.Services.Configure<AbpDistributedCacheOptions>(cacheOptions =>
        {
            cacheOptions.GlobalCacheEntryOptions.SlidingExpiration = TimeSpan.FromMinutes(20);
        });
    }
}
```

A few things to notice:

- **The open generics are singletons.** `DistributedCache<TCacheItem>` and `DistributedCache<TCacheItem, TCacheKey>` carry no per-request state — the only stateful collaborator is the underlying `IDistributedCache`, which is itself a singleton in `Microsoft.Extensions.Caching.Distributed`. Per-tenant key prefixing happens at call time inside `DistributedCacheKeyNormalizer`, not via DI scope.
- **`AddDistributedMemoryCache()` is the default backing store.** Out of the box, the typed cache is backed by `MemoryDistributedCache`, which is process-local. Modules that want a real distributed cache replace the registration — `Volo.Abp.Caching.StackExchangeRedis` does exactly that (see [StackExchange.Redis](/caching/stackexchange-redis)).
- **`AbpMultiTenancyModule` is a hard dependency.** The key normalizer resolves `ICurrentTenant` to prepend a `t:<id>` segment, so any host that uses caching also pulls in multi-tenancy. See [Multi-tenancy overview](/multitenancy/overview) for how the tenant flows.
- **`AbpJsonModule` is a hard dependency.** The default `Utf8JsonDistributedCacheSerializer` requires an `IJsonSerializer`.

## `IDistributedCache<TCacheItem>` — the typed cache

Application code never resolves `Microsoft.Extensions.Caching.Distributed.IDistributedCache` directly. Instead, it injects one of the two ABP generics:

```csharp title="framework/src/Volo.Abp.Caching/Volo/Abp/Caching/IDistributedCache.cs"
public interface IDistributedCache<TCacheItem> : IDistributedCache<TCacheItem, string>
    where TCacheItem : class
{
    IDistributedCache<TCacheItem, string> InternalCache { get; }
}
```

`IDistributedCache<TCacheItem>` is the common case — a string-keyed cache. The second generic, `IDistributedCache<TCacheItem, TCacheKey>`, lets you use any non-null type (`Guid`, a `record`, etc.) as the key; ABP calls `key.ToString()` once and feeds the result into the normalizer.

The interface surface is the same for both: a synchronous and asynchronous variant of `Get`, `GetMany`, `GetOrAdd`, `GetOrAddMany`, `Set`, `SetMany`, `Refresh`, `RefreshMany`, `Remove`, and `RemoveMany`. The signatures all share three common parameters:

- `bool? hideErrors` — overrides `AbpDistributedCacheOptions.HideErrors` on a per-call basis. When the effective value is `true`, the implementation catches the exception, forwards it to `IExceptionNotifier`, and returns `null` (or no-ops) instead of throwing.
- `bool considerUow` — when `true` and there is an [active unit of work](/uow/overview), reads are first satisfied from a per-UoW dictionary and writes are deferred to `IUnitOfWork.OnCompleted`. See [Distributed cache](/caching/distributed-cache) for the full UoW flow.
- `CancellationToken token` — falls back through `ICancellationTokenProvider` before being passed to the underlying store.

A representative `GetOrAddAsync` call site looks like this:

```csharp
public class ProductCache
{
    private readonly IDistributedCache<ProductCacheItem> _cache;

    public ProductCache(IDistributedCache<ProductCacheItem> cache)
    {
        _cache = cache;
    }

    public Task<ProductCacheItem?> GetAsync(Guid productId)
    {
        return _cache.GetOrAddAsync(
            productId.ToString(),
            async () => await LoadFromDatabaseAsync(productId),
            () => new DistributedCacheEntryOptions
            {
                SlidingExpiration = TimeSpan.FromMinutes(5)
            }
        );
    }
}
```

`GetOrAddAsync` is implemented with a double-checked lock via a `SemaphoreSlim` inside `DistributedCache<,>`, so concurrent misses won't all execute the factory.

### Convention: the `*CacheItem` suffix

`DistributedCache<TCacheItem, TCacheKey>.SetDefaultOptions()` derives a cache name from the CLR type:

```csharp title="framework/src/Volo.Abp.Caching/Volo/Abp/Caching/DistributedCache.cs"
protected virtual void SetDefaultOptions()
{
    CacheName = CacheNameAttribute.GetCacheName(typeof(TCacheItem));

    //IgnoreMultiTenancy
    IgnoreMultiTenancy = typeof(TCacheItem).IsDefined(typeof(IgnoreMultiTenancyAttribute), true);

    //Configure default cache entry options
    DefaultCacheOptions = GetDefaultCacheEntryOptions();
}
```

`CacheNameAttribute.GetCacheName(typeof(TCacheItem))` returns the value of an explicit `[CacheName]` attribute when present, otherwise it falls back to the CLR full name with the trailing `CacheItem` suffix stripped:

```csharp title="framework/src/Volo.Abp.Caching/Volo/Abp/Caching/CacheNameAttribute.cs"
public static string GetCacheName(Type cacheItemType)
{
    var cacheNameAttribute = cacheItemType
        .GetCustomAttributes(true)
        .OfType<CacheNameAttribute>()
        .FirstOrDefault();

    if (cacheNameAttribute != null)
    {
        return cacheNameAttribute.Name;
    }

    return cacheItemType.FullName!.RemovePostFix("CacheItem")!;
}
```

So `MyApp.Catalog.ProductCacheItem` becomes the cache name `MyApp.Catalog.Product`, while a type decorated with `[CacheName("products")]` becomes simply `products`. This name is later embedded in the normalized key as `c:<cacheName>` — see the key normalizer section below.

## `CacheNameAttribute`

The attribute is a single immutable property:

```csharp title="framework/src/Volo.Abp.Caching/Volo/Abp/Caching/CacheNameAttribute.cs"
[AttributeUsage(AttributeTargets.Class | AttributeTargets.Interface | AttributeTargets.Struct)]
public class CacheNameAttribute : Attribute
{
    public string Name { get; }

    public CacheNameAttribute([NotNull] string name)
    {
        Check.NotNull(name, nameof(name));

        Name = name;
    }
    // ...
}
```

Use it when:

- You want a short, human-readable cache name (e.g. `"users"`) instead of the type's full name.
- You're renaming or moving a cache-item POCO and need to preserve the on-the-wire key so existing entries are still hit.
- Two different CLR types should share the same logical cache (rare, but legal — the attribute can sit on a marker interface that both types implement).

The attribute is statically inspected once per closed generic, in `DistributedCache<TCacheItem, TCacheKey>`'s constructor (via `SetDefaultOptions`). There's no runtime overhead per call.

## `IgnoreMultiTenancyAttribute`

`IgnoreMultiTenancyAttribute` is defined in the multi-tenancy abstractions, not in the caching module, but caching is its primary client:

```csharp title="framework/src/Volo.Abp.MultiTenancy.Abstractions/Volo/Abp/MultiTenancy/IgnoreMultiTenancyAttribute.cs"
[AttributeUsage(AttributeTargets.All)]
public class IgnoreMultiTenancyAttribute : Attribute
{
}
```

When the attribute is present on the cache-item type, `DistributedCache<,>.SetDefaultOptions` sets `IgnoreMultiTenancy = true`, and the key normalizer skips the `t:<tenantId>` prefix even if a current tenant is set. This is how host-level caches (e.g. the list of tenants in `AbpTenantCacheItem`, feature definitions, permission definitions) stay shared across the whole installation instead of being duplicated per tenant.

Decorate the **cache-item type**, not the cache key:

```csharp
[IgnoreMultiTenancy]
public class FeatureDefinitionCacheItem
{
    public string Name { get; set; }
    public bool IsEnabled { get; set; }
}
```

If you also need a deterministic shared name, combine both attributes:

```csharp
[IgnoreMultiTenancy]
[CacheName("feature-definitions")]
public class FeatureDefinitionCacheItem { /* ... */ }
```

The normalized key in this case is `c:feature-definitions,k:<prefix><key>` regardless of the current tenant. For the inverse — caches that explicitly partition per tenant — simply don't apply `[IgnoreMultiTenancy]`; the normalizer reads `ICurrentTenant.Id` at call time. See [Current tenant](/multitenancy/current-tenant) for the ambient state.

## `AbpDistributedCacheOptions`

`AbpDistributedCacheOptions` is the single configuration object for the whole caching subsystem. It's a plain `IOptions<T>` target, registered via the standard pattern documented in [Options pattern](/config/options-classes).

```csharp title="framework/src/Volo.Abp.Caching/Volo/Abp/Caching/AbpDistributedCacheOptions.cs"
public class AbpDistributedCacheOptions
{
    /// <summary>
    /// Throw or hide exceptions for the distributed cache.
    /// </summary>
    public bool HideErrors { get; set; } = true;

    /// <summary>
    /// Cache key prefix.
    /// </summary>
    public string KeyPrefix { get; set; }

    /// <summary>
    /// Global Cache entry options.
    /// </summary>
    public DistributedCacheEntryOptions GlobalCacheEntryOptions { get; set; }

    /// <summary>
    /// List of all cache configurators.
    /// (func argument:Name of cache)
    /// </summary>
    public List<Func<string, DistributedCacheEntryOptions?>> CacheConfigurators { get; set; }

    public AbpDistributedCacheOptions()
    {
        CacheConfigurators = new List<Func<string, DistributedCacheEntryOptions?>>();
        GlobalCacheEntryOptions = new DistributedCacheEntryOptions();
        KeyPrefix = "";
    }
}
```

### `HideErrors`

When `true` (the default), every `Get`/`Set`/`Remove` wraps the underlying call in a try-catch, hands the exception to `Logger.LogException(..., LogLevel.Warning)` and to `IExceptionNotifier`, and returns a default value. This is critical for resilience — a cluster-wide Redis outage degrades the application to "everything is a cache miss" instead of crashing every request.

Override per-call by passing `hideErrors: false` to any of the cache APIs. That is what diagnostic code, eager warmup paths, and tests typically do.

### `KeyPrefix`

A static string prepended to every key (after the cache name segment). The most common use case is **sharing one Redis instance between multiple environments or applications** — set `KeyPrefix = "staging-"` for the staging deployment and `KeyPrefix = "prod-"` for production. See [Configuration & appsettings](/config/connection-strings-and-databases) for binding the value from `appsettings.json`.

### `GlobalCacheEntryOptions`

A `DistributedCacheEntryOptions` instance applied whenever a `Set` call doesn't pass an explicit `options` argument **and** no per-cache configurator returns a value. The `AbpCachingModule` seeds this with `SlidingExpiration = TimeSpan.FromMinutes(20)`. Override it from any consuming module:

```csharp
Configure<AbpDistributedCacheOptions>(options =>
{
    options.GlobalCacheEntryOptions = new DistributedCacheEntryOptions
    {
        AbsoluteExpirationRelativeToNow = TimeSpan.FromHours(1),
        SlidingExpiration = TimeSpan.FromMinutes(10)
    };
});
```

### `CacheConfigurators`

`CacheConfigurators` is a list of `Func<string, DistributedCacheEntryOptions?>` — each delegate receives the cache name and either returns options to apply or `null` to defer to the next configurator. `DistributedCache<,>.GetDefaultCacheEntryOptions` walks the list and returns the first non-null result, falling back to `GlobalCacheEntryOptions`:

```csharp title="framework/src/Volo.Abp.Caching/Volo/Abp/Caching/DistributedCache.cs"
protected virtual DistributedCacheEntryOptions GetDefaultCacheEntryOptions()
{
    foreach (var configure in _distributedCacheOption.CacheConfigurators)
    {
        var options = configure.Invoke(CacheName);
        if (options != null)
        {
            return options;
        }
    }

    return _distributedCacheOption.GlobalCacheEntryOptions;
}
```

There are three convenience helpers that wrap the configurator list — pick whichever expresses your intent:

```csharp title="framework/src/Volo.Abp.Caching/Volo/Abp/Caching/AbpDistributedCacheOptions.cs"
public void ConfigureCache<TCacheItem>(DistributedCacheEntryOptions? options)
{
    ConfigureCache(typeof(TCacheItem), options);
}

public void ConfigureCache(Type cacheItemType, DistributedCacheEntryOptions? options)
{
    ConfigureCache(CacheNameAttribute.GetCacheName(cacheItemType), options);
}

public void ConfigureCache(string cacheName, DistributedCacheEntryOptions? options)
{
    CacheConfigurators.Add(name => cacheName != name ? null : options);
}
```

Typical usage in a module:

```csharp
public override void ConfigureServices(ServiceConfigurationContext context)
{
    Configure<AbpDistributedCacheOptions>(options =>
    {
        // 1. Strongly-typed: matches on the CLR type's cache name.
        options.ConfigureCache<ProductCacheItem>(new DistributedCacheEntryOptions
        {
            AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(2)
        });

        // 2. By name — useful for caches owned by another module.
        options.ConfigureCache("Volo.Abp.SettingManagement.SettingCacheItem",
            new DistributedCacheEntryOptions
            {
                SlidingExpiration = TimeSpan.FromHours(6)
            });

        // 3. Custom lambda — pick options dynamically.
        options.CacheConfigurators.Add(name =>
            name.StartsWith("MyApp.Reports.")
                ? new DistributedCacheEntryOptions
                  {
                      AbsoluteExpirationRelativeToNow = TimeSpan.FromSeconds(30)
                  }
                : null);
    });
}
```

<Note>
The configurators run **once per closed generic type**, inside the `DistributedCache<,>` constructor. Changing the options after the first resolution does not retroactively affect already-constructed cache instances. Because the typed caches are singletons, this is "configure at startup" territory — there is no per-request reconfiguration story.
</Note>

## How the pieces fit together

A single `GetOrAddAsync` call traverses the following collaborators:

<Steps>
  <Step title="DI resolves the typed cache">
    `IDistributedCache<ProductCacheItem>` resolves to a singleton `DistributedCache<ProductCacheItem>`, which in turn wraps an `IDistributedCache<ProductCacheItem, string>` (also a singleton). The inner instance captured `CacheName`, `IgnoreMultiTenancy`, and `DefaultCacheOptions` at construction time.
  </Step>
  <Step title="Key normalization">
    The string key is fed into `IDistributedCacheKeyNormalizer`, which combines it with the cache name, the `KeyPrefix`, and the current tenant id (unless `IgnoreMultiTenancy` is on the item type).
  </Step>
  <Step title="UoW short-circuit (optional)">
    If `considerUow: true` and `IUnitOfWorkManager.Current` is non-null, a per-UoW dictionary keyed by the cache name is checked first. Writes go into the dictionary and are flushed by an `OnCompleted` handler when the UoW commits.
  </Step>
  <Step title="Underlying store">
    The normalized key reaches `Microsoft.Extensions.Caching.Distributed.IDistributedCache`. By default this is `MemoryDistributedCache`; with the Redis package it is `AbpRedisCache`.
  </Step>
  <Step title="Serialization">
    Bytes flow through `IDistributedCacheSerializer`. The default implementation `Utf8JsonDistributedCacheSerializer` calls `IJsonSerializer` (the abstraction documented in [JSON serialization](/serialization/overview)) and runs the result through `Encoding.UTF8`.
  </Step>
  <Step title="Exception handling">
    Any exception from the store is caught when `HideErrors` is true, logged at `Warning`, and forwarded to `IExceptionNotifier` inside a fresh service scope produced by the injected `IServiceScopeFactory`.
  </Step>
</Steps>
