ABP gives MVC and API authors a pair of base classes &mdash; `AbpController`
and `AbpControllerBase` &mdash; that pre-wire the framework's most common
dependencies through `IAbpLazyServiceProvider`. The matching
`AbpViewComponent` does the same for view components. They all live in
`framework/src/Volo.Abp.AspNetCore.Mvc/Volo/Abp/AspNetCore/Mvc/` and let
you skip the constructor noise of resolving the current user, the
localizer, an object mapper, or the logger on every controller.

## When to use which base class

| Base class | Inherits from | Use for |
| --- | --- | --- |
| `AbpController` | `Microsoft.AspNetCore.Mvc.Controller` | MVC controllers that return views, `RedirectResult`, or mix views with JSON |
| `AbpControllerBase` | `Microsoft.AspNetCore.Mvc.ControllerBase` | API/REST controllers that never return views |
| `AbpViewComponent` | `Microsoft.AspNetCore.Mvc.ViewComponent` | View components that need ABP services |

Both controller bases implement `IAvoidDuplicateCrossCuttingConcerns`, so
ABP's interceptor pipeline knows not to apply the same audit/UoW aspect
twice when the action is invoked through dynamic-proxy clients.

## Lazy service provider

`IAbpLazyServiceProvider` is the key. Every helper property is implemented
as `LazyServiceProvider.LazyGetRequiredService<T>()`, so the underlying
service is only resolved the first time the property is read and reused
within the request.

```csharp title="framework/src/Volo.Abp.AspNetCore.Mvc/Volo/Abp/AspNetCore/Mvc/AbpController.cs"
public abstract class AbpController : Controller, IAvoidDuplicateCrossCuttingConcerns
{
    public IAbpLazyServiceProvider LazyServiceProvider { get; set; } = default!;

    [Obsolete("Use LazyServiceProvider instead.")]
    public IServiceProvider ServiceProvider { get; set; } = default!;

    protected IUnitOfWorkManager UnitOfWorkManager =>
        LazyServiceProvider.LazyGetRequiredService<IUnitOfWorkManager>();

    protected IObjectMapper ObjectMapper => LazyServiceProvider.LazyGetService<IObjectMapper>(provider =>
        ObjectMapperContext == null
            ? provider.GetRequiredService<IObjectMapper>()
            : (IObjectMapper)provider.GetRequiredService(typeof(IObjectMapper<>).MakeGenericType(ObjectMapperContext)));

    protected IGuidGenerator GuidGenerator => LazyServiceProvider.LazyGetService<IGuidGenerator>(SimpleGuidGenerator.Instance);
    protected ILoggerFactory LoggerFactory => LazyServiceProvider.LazyGetRequiredService<ILoggerFactory>();
    protected ILogger Logger => LazyServiceProvider.LazyGetService<ILogger>(provider =>
        LoggerFactory?.CreateLogger(GetType().FullName!) ?? NullLogger.Instance);
    protected ICurrentUser CurrentUser => LazyServiceProvider.LazyGetRequiredService<ICurrentUser>();
    protected ICurrentTenant CurrentTenant => LazyServiceProvider.LazyGetRequiredService<ICurrentTenant>();
    protected IAuthorizationService AuthorizationService => LazyServiceProvider.LazyGetRequiredService<IAuthorizationService>();
    protected IUnitOfWork? CurrentUnitOfWork => UnitOfWorkManager?.Current;
    protected IClock Clock => LazyServiceProvider.LazyGetRequiredService<IClock>();
    protected IModelStateValidator ModelValidator => LazyServiceProvider.LazyGetRequiredService<IModelStateValidator>();
    protected IFeatureChecker FeatureChecker => LazyServiceProvider.LazyGetRequiredService<IFeatureChecker>();
    protected IAppUrlProvider AppUrlProvider => LazyServiceProvider.LazyGetRequiredService<IAppUrlProvider>();
    protected IStringLocalizerFactory StringLocalizerFactory => LazyServiceProvider.LazyGetRequiredService<IStringLocalizerFactory>();
}
```

`LazyServiceProvider` is a property-injected service set up by ABP's
property-injection registrar. You never assign it yourself.

## Available helpers

| Property | Returns | Lives in |
| --- | --- | --- |
| `CurrentUser` | `ICurrentUser` | See [Authentication](/auth) |
| `CurrentTenant` | `ICurrentTenant` | See [Multi-tenancy](/multitenancy) |
| `AuthorizationService` | `IAuthorizationService` (ASP.NET Core) | See [Authorization](/authz) |
| `UnitOfWorkManager` / `CurrentUnitOfWork` | `IUnitOfWorkManager` / `IUnitOfWork?` | Ambient UoW |
| `Clock` | `IClock` | UTC vs Local-aware clock |
| `GuidGenerator` | `IGuidGenerator` | Sequential GUID generator |
| `LoggerFactory` / `Logger` | `ILoggerFactory` / `ILogger` | Logger lazily named after the controller type |
| `FeatureChecker` | `IFeatureChecker` | Per-tenant features |
| `ObjectMapper` | `IObjectMapper` (typed via `ObjectMapperContext`) | DTO ⇆ entity mapping |
| `ModelValidator` | `IModelStateValidator` | Used by `ValidateModel()` |
| `StringLocalizerFactory` | `IStringLocalizerFactory` | Backs the `L` localizer |
| `AppUrlProvider` (controller only) | `IAppUrlProvider` | Used for safe redirects |

`ObjectMapperContext` lets you scope the mapper to a specific
configuration. Setting it triggers
`IObjectMapper<TContext>` resolution.

```csharp
public class BooksController : AbpController
{
    public BooksController()
    {
        ObjectMapperContext = typeof(BookStoreApplicationModule);
        LocalizationResource = typeof(BookStoreResource);
    }
}
```

## The `L` localizer property

Both controller bases expose a lazily-built `IStringLocalizer L` keyed by
the `LocalizationResource` type. Reassigning `LocalizationResource`
invalidates the cached localizer.

```csharp title="AbpController.cs (excerpt)"
protected IStringLocalizer L {
    get {
        if (_localizer == null)
        {
            _localizer = CreateLocalizer();
        }

        return _localizer;
    }
}
private IStringLocalizer? _localizer;

protected Type? LocalizationResource {
    get => _localizationResource;
    set {
        _localizationResource = value;
        _localizer = null;
    }
}
private Type? _localizationResource = typeof(DefaultResource);

protected virtual IStringLocalizer CreateLocalizer()
{
    if (LocalizationResource != null)
    {
        return StringLocalizerFactory.Create(LocalizationResource);
    }

    var localizer = StringLocalizerFactory.CreateDefaultOrNull();
    if (localizer == null)
    {
        throw new AbpException($"Set {nameof(LocalizationResource)} or define the default localization resource type (by configuring the {nameof(AbpLocalizationOptions)}.{nameof(AbpLocalizationOptions.DefaultResourceType)}) to be able to use the {nameof(L)} object!");
    }

    return localizer;
}
```

If you have not set `LocalizationResource` **and** the host has not
configured `AbpLocalizationOptions.DefaultResourceType`, reading `L` throws
an `AbpException`. The error message tells you exactly what to do.

## Safe redirects (AbpController only)

`AbpController` adds two methods that prevent open-redirect attacks by
falling back to the application home URL when the target is neither local
nor explicitly allow-listed:

```csharp title="AbpController.cs"
protected virtual RedirectResult RedirectSafely(string returnUrl, string? returnUrlHash = null)
{
    return Redirect(GetRedirectUrl(returnUrl, returnUrlHash));
}

protected virtual string NormalizeReturnUrl(string returnUrl)
{
    if (returnUrl.IsNullOrEmpty())
    {
        return GetAppHomeUrl();
    }

    if (Url.IsLocalUrl(returnUrl) || AppUrlProvider.IsRedirectAllowedUrl(returnUrl))
    {
        return returnUrl;
    }

    return GetAppHomeUrl();
}
```

The allow list comes from `IAppUrlProvider`, which the navigation module
populates from `AppUrlOptions.RedirectAllowedUrls`.

## ValidateModel

Both bases ship a `ValidateModel()` helper that delegates to
`IModelStateValidator`. Inside ABP's MVC integration this is also driven
automatically through `AbpAspNetCoreMvcOptions.AutoModelValidation = true`,
which is the default:

```csharp title="AbpControllerBase.cs"
protected virtual void ValidateModel()
{
    ModelValidator?.Validate(ModelState);
}
```

## AbpControllerBase: API-first base

`AbpControllerBase` is a near-copy of `AbpController` minus the `IUrlHelper`
and `RedirectSafely` helpers (which are MVC-only). Use it for REST
endpoints exposed by auto-API or hand-written API controllers.

```csharp title="framework/src/Volo.Abp.AspNetCore.Mvc/Volo/Abp/AspNetCore/Mvc/AbpControllerBase.cs"
public abstract class AbpControllerBase : ControllerBase, IAvoidDuplicateCrossCuttingConcerns
{
    public IAbpLazyServiceProvider LazyServiceProvider { get; set; } = default!;

    protected IUnitOfWorkManager UnitOfWorkManager => LazyServiceProvider.LazyGetRequiredService<IUnitOfWorkManager>();
    protected ICurrentUser CurrentUser => LazyServiceProvider.LazyGetRequiredService<ICurrentUser>();
    protected ICurrentTenant CurrentTenant => LazyServiceProvider.LazyGetRequiredService<ICurrentTenant>();
    protected IAuthorizationService AuthorizationService => LazyServiceProvider.LazyGetRequiredService<IAuthorizationService>();
    protected IFeatureChecker FeatureChecker => LazyServiceProvider.LazyGetRequiredService<IFeatureChecker>();
    // ... same logger, clock, guid generator, localizer, ModelValidator
}
```

`AbpControllerBase` is what conventional/auto controllers transparently
inherit from when you expose an `IApplicationService` as an HTTP endpoint
(see [Auto API controllers](/web/auto-api-controllers)).

## AbpViewComponent

View components need much less plumbing because they have no claim on the
ambient unit of work or the model state. Only the lazy service provider,
`ObjectMapper`, and `ObjectMapperContext` are pre-wired.

```csharp title="framework/src/Volo.Abp.AspNetCore.Mvc/Volo/Abp/AspNetCore/Mvc/AbpViewComponent.cs"
public abstract class AbpViewComponent : ViewComponent
{
    public IAbpLazyServiceProvider LazyServiceProvider { get; set; } = default!;

    [Obsolete("Use LazyServiceProvider instead.")]
    public IServiceProvider ServiceProvider { get; set; } = default!;

    protected Type? ObjectMapperContext { get; set; }

    protected IObjectMapper ObjectMapper => LazyServiceProvider.LazyGetService<IObjectMapper>(provider =>
        ObjectMapperContext == null
            ? provider.GetRequiredService<IObjectMapper>()
            : (IObjectMapper)provider.GetRequiredService(typeof(IObjectMapper<>).MakeGenericType(ObjectMapperContext)));
}
```

ASP.NET Core's `ViewComponent` already exposes `User`, `HttpContext`, and
`ViewContext`, so you compose ABP services on top of them as needed.

## How property injection actually happens

`AbpAspNetCoreMvcModule` registers controllers and view components as
services so DI can fill the `LazyServiceProvider` property:

```csharp title="AbpAspNetCoreMvcModule.cs (excerpt)"
//Use DI to create controllers
mvcBuilder.AddControllersAsServices();

//Use DI to create view components
mvcBuilder.AddViewComponentsAsServices();

//Use DI to create razor page
context.Services.Replace(ServiceDescriptor.Singleton<IPageModelActivatorProvider, ServiceBasedPageModelActivatorProvider>());
```

That replacement of `IPageModelActivatorProvider` is what lets Razor Pages
also receive `LazyServiceProvider` through property injection.

## Putting it together

A typical ABP REST controller looks like this:

```csharp
[Authorize]
[Route("api/app/books")]
public class BooksController : AbpControllerBase
{
    public BooksController()
    {
        LocalizationResource = typeof(BookStoreResource);
    }

    [HttpGet("{id}")]
    public async Task<BookDto> GetAsync(Guid id)
    {
        Logger.LogDebug("Loading book {Id}", id);

        if (!await AuthorizationService.IsGrantedAsync(BookPermissions.Books.Read))
        {
            throw new AbpAuthorizationException(L["YouAreNotAllowedToReadBooks"]);
        }

        var book = await _bookRepository.GetAsync(id);
        return ObjectMapper.Map<Book, BookDto>(book);
    }
}
```

Notice you never inject `ICurrentUser`, `ILogger`, `IObjectMapper`,
`IStringLocalizerFactory`, or `IAuthorizationService` &mdash; they are
exposed as lazy properties. Concrete repositories or domain services
remain constructor-injected so the dependency intent stays explicit.

For controllers that get **generated** from `IRemoteService` interfaces by
ABP convention, continue with [Auto API controllers](/web/auto-api-controllers).
