`Volo.Abp.MongoDB` is ABP's first-party integration with the official **MongoDB C# driver**. It mirrors the [EF Core integration](/data/entity-framework-core) — same `IRepository<TEntity, TKey>` surface, same unit-of-work model, same `[ConnectionStringName]` convention — but talks `BsonDocument`s instead of SQL. Every solution template generated by the ABP CLI ships both an `EntityFrameworkCore` and a `MongoDB` project, so swapping providers (for one bounded context or the whole app) is a question of which `[DependsOn]` you keep. This page walks every file in the package and shows how `AbpMongoDbContext`, `MongoDbRepository<,,>`, and `UnitOfWorkMongoDbContextProvider<>` compose.

## The module

```csharp framework/src/Volo.Abp.MongoDB/Volo/Abp/MongoDB/AbpMongoDbModule.cs
[DependsOn(typeof(AbpDddDomainModule))]
public class AbpMongoDbModule : AbpModule
{
    public override void PreConfigureServices(ServiceConfigurationContext context)
    {
        context.Services.AddConventionalRegistrar(new AbpMongoDbConventionalRegistrar());
    }

    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        context.Services.TryAddTransient(
            typeof(IMongoDbContextProvider<>),
            typeof(UnitOfWorkMongoDbContextProvider<>)
        );

        context.Services.TryAddTransient(
            typeof(IMongoDbRepositoryFilterer<>),
            typeof(MongoDbRepositoryFilterer<>)
        );

        context.Services.TryAddTransient(
            typeof(IMongoDbRepositoryFilterer<,>),
            typeof(MongoDbRepositoryFilterer<,>)
        );

        context.Services.AddTransient(
            typeof(IMongoDbContextEventOutbox<>),
            typeof(MongoDbContextEventOutbox<>)
        );

        context.Services.AddTransient(
            typeof(IMongoDbContextEventInbox<>),
            typeof(MongoDbContextEventInbox<>)
        );
    }
}
```

Three responsibilities:

1. **Conventional registration** — `AbpMongoDbConventionalRegistrar` wires DbContexts and their derived classes as `ITransientDependency` instances even when they do not implement the marker interface, because they inherit from `AbpMongoDbContext` (which does).
2. **UoW provider** — every `IMongoDbContextProvider<MyMongoDbContext>` resolves to `UnitOfWorkMongoDbContextProvider<MyMongoDbContext>`.
3. **Outbox/inbox** — the distributed-events tables.

## `AbpMongoDbContext`

`AbpMongoDbContext` is the *abstract base* class your context inherits from. Unlike EF Core, MongoDB has no `OnModelCreating(ModelBuilder)`, so `CreateModel(IMongoModelBuilder)` plays the same role:

```csharp framework/src/Volo.Abp.MongoDB/Volo/Abp/MongoDB/AbpMongoDbContext.cs
public abstract class AbpMongoDbContext : IAbpMongoDbContext, ITransientDependency
{
    public IAbpLazyServiceProvider LazyServiceProvider { get; set; } = default!;

    public IMongoModelSource ModelSource { get; set; } = default!;

    public IMongoClient Client { get; private set; } = default!;

    public IMongoDatabase Database { get; private set; } = default!;

    public IClientSessionHandle? SessionHandle { get; private set; }

    protected internal virtual void CreateModel(IMongoModelBuilder modelBuilder)
    {

    }

    public virtual void InitializeDatabase(IMongoDatabase database, IMongoClient client, IClientSessionHandle? sessionHandle)
    {
        Database = database;
        Client = client;
        SessionHandle = sessionHandle;
    }

    public virtual IMongoCollection<T> Collection<T>()
    {
        return Database.GetCollection<T>(GetCollectionName<T>());
    }

    public virtual void InitializeCollections(IMongoDatabase database)
    {
        Database = database;
        ModelSource.GetModel(this);
    }

    protected virtual string GetCollectionName<T>()
    {
        return GetEntityModel<T>().CollectionName;
    }

    protected virtual IMongoEntityModel GetEntityModel<TEntity>()
    {
        var model = ModelSource.GetModel(this).Entities.GetOrDefault(typeof(TEntity));

        if (model == null)
        {
            throw new AbpException("Could not find a model for given entity type: " + typeof(TEntity).AssemblyQualifiedName);
        }

        return model;
    }
}
```

A typical context override looks like:

```csharp BookStoreMongoDbContext.cs
[ConnectionStringName("BookStore")]
public class BookStoreMongoDbContext : AbpMongoDbContext, IBookStoreMongoDbContext
{
    public IMongoCollection<Book> Books => Collection<Book>();

    protected override void CreateModel(IMongoModelBuilder modelBuilder)
    {
        base.CreateModel(modelBuilder);
        modelBuilder.Entity<Book>(b => b.CollectionName = "Books");
    }
}
```

`InitializeDatabase` is called by `UnitOfWorkMongoDbContextProvider` with the live `IMongoClient`, `IMongoDatabase`, and (if a transaction is active) the `IClientSessionHandle`. `SessionHandle` is what makes calls like `collection.InsertOneAsync(SessionHandle, doc)` participate in the active transaction.

## `AbpMongoDbContextOptions`

```csharp framework/src/Volo.Abp.MongoDB/Volo/Abp/MongoDB/AbpMongoDbContextOptions.cs
public class AbpMongoDbContextOptions
{
    internal Dictionary<MultiTenantDbContextType, Type> DbContextReplacements { get; }

    public Action<MongoClientSettings>? MongoClientSettingsConfigurer { get; set; }

    public AbpMongoDbContextOptions()
    {
        DbContextReplacements = new Dictionary<MultiTenantDbContextType, Type>();
    }

    internal Type GetReplacedTypeOrSelf(Type dbContextType, MultiTenancySides multiTenancySides = MultiTenancySides.Both)
    {
        var replacementType = dbContextType;
        while (true)
        {
            var foundType = DbContextReplacements.LastOrDefault(x => x.Key.Type == replacementType && x.Key.MultiTenancySide.HasFlag(multiTenancySides));
            if (!foundType.Equals(default(KeyValuePair<MultiTenantDbContextType, Type>)))
            {
                if (foundType.Value == dbContextType)
                {
                    throw new AbpException(
                        "Circular DbContext replacement found for " +
                        dbContextType.AssemblyQualifiedName
                    );
                }
                replacementType = foundType.Value;
            }
            else
            {
                return replacementType;
            }
        }
    }
}
```

Two responsibilities:

- **`DbContextReplacements`** — the `[ReplaceDbContext]` graph (e.g. your `BookStoreMongoDbContext` replaces `IIdentityMongoDbContext`).
- **`MongoClientSettingsConfigurer`** — global callback applied to `MongoClientSettings` before the driver builds the client. Use it to tweak `MaxConnectionPoolSize`, register a `ClusterConfigurator`, or set TLS callbacks:

```csharp BookStoreMongoDbContextRegistrationModule.cs
Configure<AbpMongoDbContextOptions>(options =>
{
    options.MongoClientSettingsConfigurer = settings =>
    {
        settings.MaxConnectionPoolSize = 200;
        settings.UseTls = true;
    };
});
```

## `IMongoDbContextProvider<TMongoDbContext>`

```csharp framework/src/Volo.Abp.MongoDB/Volo/Abp/MongoDB/IMongoDbContextProvider.cs
public interface IMongoDbContextProvider<TMongoDbContext>
    where TMongoDbContext : IAbpMongoDbContext
{
    [Obsolete("Use CreateDbContextAsync")]
    TMongoDbContext GetDbContext();

    Task<TMongoDbContext> GetDbContextAsync(CancellationToken cancellationToken = default);
}
```

`UnitOfWorkMongoDbContextProvider<TMongoDbContext>` is the default implementation. It mirrors the EF Core variant: resolve the target context type (honouring `[ReplaceDbContext]`), resolve the connection string via `IConnectionStringResolver`, key the UoW slot by `{type}_{connectionString}`, and either return the cached instance or build a fresh one:

```csharp framework/src/Volo.Abp.MongoDB/Volo/Abp/Uow/MongoDB/UnitOfWorkMongoDbContextProvider.cs
public virtual async Task<TMongoDbContext> GetDbContextAsync(CancellationToken cancellationToken = default)
{
    var unitOfWork = UnitOfWorkManager.Current;
    if (unitOfWork == null)
    {
        throw new AbpException(
            $"A {nameof(IMongoDatabase)} instance can only be created inside a unit of work!");
    }

    var targetDbContextType = DbContextTypeProvider.GetDbContextType(typeof(TMongoDbContext));
    var connectionString = await ResolveConnectionStringAsync(targetDbContextType);
    var dbContextKey = $"{targetDbContextType.FullName}_{connectionString}";
```

Inside `CreateDbContext`, the MongoDB driver does not natively cache `IMongoClient` instances, so ABP stamps the database name from `MongoUrl.DatabaseName` (or falls back to the `[ConnectionStringName]` value).

## `AbpMongoDbConventionalRegistrar`

The conventional registrar is what makes `AbpMongoDbContext`-derived classes show up in DI without an explicit `services.AddTransient`. It also auto-registers everything implementing `IMongoDbContext` interfaces declared on the user's context — this is how an injected `IBookStoreMongoDbContext` resolves to `BookStoreMongoDbContext`.

## `MongoDbRepository<TMongoDbContext, TEntity, TKey>`

```csharp framework/src/Volo.Abp.MongoDB/Volo/Abp/Domain/Repositories/MongoDB/MongoDbRepository.cs
public class MongoDbRepository<TMongoDbContext, TEntity>
    : RepositoryBase<TEntity>,
    IMongoDbRepository<TEntity>
    where TMongoDbContext : IAbpMongoDbContext
    where TEntity : class, IEntity
{
    public virtual async Task<IMongoCollection<TEntity>> GetCollectionAsync(CancellationToken cancellationToken = default)
    {
        return (await GetDbContextAsync(GetCancellationToken(cancellationToken))).Collection<TEntity>();
    }

    public virtual async Task<IMongoDatabase> GetDatabaseAsync(CancellationToken cancellationToken = default)
    {
        return (await GetDbContextAsync(GetCancellationToken(cancellationToken))).Database;
    }

    protected async Task<IClientSessionHandle?> GetSessionHandleAsync(CancellationToken cancellationToken = default)
    {
        return (await GetDbContextAsync(GetCancellationToken(cancellationToken))).SessionHandle;
    }

    protected Task<TMongoDbContext> GetDbContextAsync(CancellationToken cancellationToken = default)
    {
        cancellationToken = GetCancellationToken(cancellationToken);

        // Multi-tenancy unaware entities should always use the host connection string
        if (!EntityHelper.IsMultiTenant<TEntity>())
        {
            using (CurrentTenant.Change(null))
            {
                return DbContextProvider.GetDbContextAsync(cancellationToken);
            }
        }

        return DbContextProvider.GetDbContextAsync(cancellationToken);
    }
}
```

Two things to notice: the same `CurrentTenant.Change(null)` trick the EF Core repository uses to keep host entities on the host connection string, and the fact that every operation funnels through `GetDbContextAsync` — `await GetCollectionAsync()` is just a thin wrapper. Inserts include the active session handle so writes participate in transactions:

```csharp framework/src/Volo.Abp.MongoDB/Volo/Abp/Domain/Repositories/MongoDB/MongoDbRepository.cs
var dbContext = await GetDbContextAsync(cancellationToken);
var collection = dbContext.Collection<TEntity>();

if (dbContext.SessionHandle != null)
{
    // SessionHandle is non-null when a multi-document transaction is open
    // ... InsertOneAsync(dbContext.SessionHandle, entity, …)
```

## Queryable interception

The MongoDB C# driver returns `IMongoQueryable<T>` from `Collection.AsQueryable()`. Castle DynamicProxy interception (used by ABP's auditing, validation, and unit-of-work interceptors) wraps repository instances, so the queryable that flows out of a repository call is *wrapped*. `MongoDbAsyncQueryableProvider` therefore calls `ProxyHelper.UnProxy` before casting, so `ToListAsync` / `FirstOrDefaultAsync` extension methods continue to resolve to the MongoDB-aware overloads:

```csharp framework/src/Volo.Abp.MongoDB/Volo/Abp/MongoDB/MongoDbAsyncQueryableProvider.cs
public class MongoDbAsyncQueryableProvider : IAsyncQueryableProvider, ISingletonDependency
{
    public bool CanExecute<T>(IQueryable<T> queryable)
    {
        return ProxyHelper.UnProxy(queryable) is IMongoQueryable<T>;
    }

    protected virtual IMongoQueryable<T> GetMongoQueryable<T>(IQueryable<T> queryable)
    {
        return ProxyHelper.UnProxy(queryable).As<IMongoQueryable<T>>();
    }
}
```

This is the equivalent of EF Core's `EfCoreAsyncQueryableProvider` — see [Property Injection and Interception](/di/property-injection-and-interception) for the surrounding dynamic-proxy story.

## Registering a MongoDb context

`AddMongoDbContext<TMongoDbContext>` is the registration call your `*MongoDbModule` makes:

```csharp framework/src/Volo.Abp.MongoDB/Microsoft/Extensions/DependencyInjection/AbpMongoDbServiceCollectionExtensions.cs
public static IServiceCollection AddMongoDbContext<TMongoDbContext>(this IServiceCollection services, Action<IAbpMongoDbContextRegistrationOptionsBuilder>? optionsBuilder = null)
    where TMongoDbContext : AbpMongoDbContext
{
    var options = new AbpMongoDbContextRegistrationOptions(typeof(TMongoDbContext), services);

    var replacedDbContextTypes = typeof(TMongoDbContext).GetCustomAttributes<ReplaceDbContextAttribute>(true)
        .SelectMany(x => x.ReplacedDbContextTypes).ToList();

    foreach (var dbContextType in replacedDbContextTypes)
    {
        options.ReplaceDbContext(dbContextType.Type, multiTenancySides: dbContextType.MultiTenancySide);
    }

    optionsBuilder?.Invoke(options);

    foreach (var entry in options.ReplacedDbContextTypes)
    {
        var originalDbContextType = entry.Key.Type;
        var targetDbContextType = entry.Value ?? typeof(TMongoDbContext);

        services.Replace(ServiceDescriptor.Transient(originalDbContextType, sp =>
        {
            var dbContextType = sp.GetRequiredService<IMongoDbContextTypeProvider>().GetDbContextType(originalDbContextType);
            return sp.GetRequiredService(dbContextType);
        }));

        services.Configure<AbpMongoDbContextOptions>(opts =>
        {
            var multiTenantDbContextType = new MultiTenantDbContextType(originalDbContextType, entry.Key.MultiTenancySide);
            opts.DbContextReplacements[multiTenantDbContextType] = targetDbContextType;
        });
    }

    new MongoDbRepositoryRegistrar(options).AddRepositories();

    return services;
}
```

Two responsibilities mirror the EF Core code: process `[ReplaceDbContext]` and run `MongoDbRepositoryRegistrar.AddRepositories()` which scans `options.CustomRepositories` plus, when `IncludeAllEntities = true`, every `IEntity` declared on the context.

## End-to-end flow

```mermaid
sequenceDiagram
    autonumber
    participant App as Application Service
    participant Repo as MongoDbRepository&lt;Book,Guid&gt;
    participant Prov as UnitOfWorkMongoDbContextProvider&lt;BookStoreMongoDbContext&gt;
    participant Res as IConnectionStringResolver
    participant UoW as IUnitOfWork
    participant Ctx as BookStoreMongoDbContext
    participant Mongo as MongoDB driver

    App->>Repo: await _bookRepo.InsertAsync(book)
    Repo->>Prov: GetDbContextAsync()
    Prov->>UoW: UnitOfWorkManager.Current
    Prov->>Res: ResolveAsync(targetType)
    Prov->>UoW: FindDatabaseApi(key)
    alt not cached
        Prov->>Ctx: new BookStoreMongoDbContext()
        Prov->>Mongo: MongoClient (apply MongoClientSettingsConfigurer)
        Mongo-->>Prov: IMongoDatabase
        Prov->>Ctx: InitializeDatabase(...)
        Prov->>UoW: AddDatabaseApi(key, MongoDbDatabaseApi(ctx))
    end
    Prov-->>Repo: BookStoreMongoDbContext
    Repo->>Ctx: Collection&lt;Book&gt;()
    Repo->>Mongo: InsertOneAsync(SessionHandle?, book)
    App->>UoW: CompleteAsync()
    UoW->>Mongo: SessionHandle.CommitTransaction()
```
