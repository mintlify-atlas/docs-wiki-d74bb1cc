The `Volo.Abp.Http` and `Volo.Abp.Http.Abstractions` packages of the **ABP Framework** define the wire-level contracts that every ABP HTTP client and server speak. They are intentionally tiny: a constants class, a conventions helper for HTTP verbs, the `ApplicationApiDescriptionModel` graph (what `/api/abp/api-definition` returns), the `RemoteServiceErrorInfo` family (what an ABP server returns when something goes wrong), and the jQuery proxy script generator that emits client-side JavaScript for legacy MVC views.

<Note>
`Volo.Abp.Http.Abstractions` only depends on `Volo.Abp.Core` (via `AbpModule`). It is safe to reference from front-end UI projects that don't want a transitive `HttpClient` dependency.
</Note>

## Module graph

`AbpHttpModule` (in `Volo.Abp.Http`) depends on `AbpHttpAbstractionsModule`, `AbpJsonModule` and `AbpMinifyModule`. Its only `ConfigureServices` body registers the jQuery proxy generator:

```csharp title="framework/src/Volo.Abp.Http/Volo/Abp/Http/AbpHttpModule.cs"
[DependsOn(typeof(AbpHttpAbstractionsModule))]
[DependsOn(typeof(AbpJsonModule))]
[DependsOn(typeof(AbpMinifyModule))]
public class AbpHttpModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpApiProxyScriptingOptions>(options =>
        {
            options.Generators[JQueryProxyScriptGenerator.Name] = typeof(JQueryProxyScriptGenerator);
        });
    }
}
```

`AbpHttpAbstractionsModule` is empty — it exists purely as a dependency anchor.

## Constants

```csharp title="framework/src/Volo.Abp.Http/Volo/Abp/Http/AbpHttpConsts.cs"
namespace Volo.Abp.Http;

public static class AbpHttpConsts
{
    public const string AbpErrorFormat = "_AbpErrorFormat";
    public const string AbpTenantResolveError = "Abp-Tenant-Resolve-Error";
}
```

- `_AbpErrorFormat` — set as a *response* header by the server when the body is an ABP `RemoteServiceErrorResponse`. The dynamic-proxy client uses it as the discriminator that tells deserialisation what type to expect.
- `Abp-Tenant-Resolve-Error` — set when ABP fails to resolve a tenant header on the request (e.g. unknown tenant). Useful for UIs that want to invalidate cached tenant state.

## `HttpMethodHelper` — conventional verb resolution

```csharp title="framework/src/Volo.Abp.Http/Volo/Abp/Http/HttpMethodHelper.cs"
public static class HttpMethodHelper
{
    public const string DefaultHttpVerb = "POST";

    public static Dictionary<string, string[]> ConventionalPrefixes { get; set; } = new Dictionary<string, string[]>
        {
            {"GET", new[] {"GetList", "GetAll", "Get"}},
            {"PUT", new[] {"Put", "Update"}},
            {"DELETE", new[] {"Delete", "Remove"}},
            {"POST", new[] {"Create", "Add", "Insert", "Post"}},
            {"PATCH", new[] {"Patch"}}
        };

    public static string GetConventionalVerbForMethodName(string methodName) { /* ... */ }
    public static string RemoveHttpMethodPrefix(string methodName, string httpMethod) { /* ... */ }
    public static HttpMethod ConvertToHttpMethod(string? httpMethod) { /* ... */ }
}
```

This is the **single source of truth** for ABP's auto-controller convention: a method called `UpdateUserAsync` becomes `PUT` and the route segment loses `Update`. The dictionary is mutable — you can add or replace prefixes from your own module before any conventions run.

`ConvertToHttpMethod` throws `AbpException("Unknown HTTP METHOD: ...")` for anything outside the eight verbs it knows about (GET/POST/PUT/DELETE/OPTIONS/TRACE/HEAD/PATCH).

## API description model

`/api/abp/api-definition` returns an `ApplicationApiDescriptionModel`. The dynamic proxy uses it to discover routes, parameters and HTTP verbs at runtime; tooling uses it to generate static proxies.

```csharp title="framework/src/Volo.Abp.Http/Volo/Abp/Http/Modeling/ApplicationApiDescriptionModel.cs"
[Serializable]
public class ApplicationApiDescriptionModel
{
    public IDictionary<string, ModuleApiDescriptionModel> Modules { get; set; } = default!;

    public IDictionary<string, TypeApiDescriptionModel> Types { get; set; } = default!;

    public static ApplicationApiDescriptionModel Create() { /* ... */ }

    public ModuleApiDescriptionModel AddModule(ModuleApiDescriptionModel module) { /* ... */ }

    public ModuleApiDescriptionModel GetOrAddModule(string rootPath, string remoteServiceName) { /* ... */ }

    public ApplicationApiDescriptionModel CreateSubModel(string[]? modules = null, string[]? controllers = null, string[]? actions = null) { /* ... */ }
}
```

The shape is hierarchical: `Application -> Module -> Controller -> Action -> Parameter`. Each `ActionApiDescriptionModel` carries `HttpMethod`, `Url`, `SupportedVersions`, `AllowAnonymous` and the parameter list. `Types` is a flat type dictionary keyed by full type name; it lets DTO definitions be shared across actions without duplication.

### `AbpApiDescriptionModelOptions`

```csharp title="framework/src/Volo.Abp.Http.Abstractions/Volo/Abp/Http/Modeling/AbpApiDescriptionModelOptions.cs"
public class AbpApiDescriptionModelOptions
{
    public HashSet<Type> IgnoredInterfaces { get; }

    public AbpApiDescriptionModelOptions()
    {
        IgnoredInterfaces = new HashSet<Type>
            {
                typeof(ITransientDependency),
                typeof(ISingletonDependency),
                typeof(IDisposable),
                typeof(IAvoidDuplicateCrossCuttingConcerns)
            };
    }
}
```

The provider that builds an `ApplicationApiDescriptionModel` walks every interface on a service type and skips anything in `IgnoredInterfaces` so DI markers don't leak into the API definition. Add your own marker interfaces here if you build cross-cutting interfaces that should not appear.

## The error contract

```csharp title="framework/src/Volo.Abp.ExceptionHandling/Volo/Abp/Http/RemoteServiceErrorInfo.cs"
[Serializable]
public class RemoteServiceErrorInfo
{
    public string? Code { get; set; }
    public string? Message { get; set; }
    public string? Details { get; set; }
    public IDictionary? Data { get; set; }
    public RemoteServiceValidationErrorInfo[]? ValidationErrors { get; set; }

    public RemoteServiceErrorInfo() { }

    public RemoteServiceErrorInfo(string message, string? details = null, string? code = null, IDictionary? data = null)
    {
        Message = message;
        Details = details;
        Code = code;
        Data = data;
    }
}
```

`RemoteServiceErrorResponse` is just a one-property envelope:

```csharp title="framework/src/Volo.Abp.ExceptionHandling/Volo/Abp/Http/RemoteServiceErrorResponse.cs"
public class RemoteServiceErrorResponse
{
    public RemoteServiceErrorInfo Error { get; set; }

    public RemoteServiceErrorResponse(RemoteServiceErrorInfo error)
    {
        Error = error;
    }
}
```

`ValidationErrors` carries field-level details, modelled on the standard ABP validation pipeline:

```csharp title="framework/src/Volo.Abp.ExceptionHandling/Volo/Abp/Http/RemoteServiceValidationErrorInfo.cs"
[Serializable]
public class RemoteServiceValidationErrorInfo
{
    public string Message { get; set; } = default!;
    public string[] Members { get; set; } = default!;

    public RemoteServiceValidationErrorInfo() { }
    public RemoteServiceValidationErrorInfo(string message) { Message = message; }
    public RemoteServiceValidationErrorInfo(string message, string[] members) : this(message) { Members = members; }
    public RemoteServiceValidationErrorInfo(string message, string member) : this(message, new[] { member }) { }
}
```

A typical 400 response body therefore looks like:

```json
{
  "error": {
    "code": "Volo.Abp.Identity:010002",
    "message": "Your request is not valid!",
    "details": null,
    "data": null,
    "validationErrors": [
      { "message": "The Name field is required.", "members": ["name"] }
    ]
  }
}
```

## The JS proxy generator

`AbpHttpModule` registers exactly one generator out of the box: `JQueryProxyScriptGenerator`. It is exposed by the constant `JQueryProxyScriptGenerator.Name = "jquery"`. Custom generators (TypeScript, etc.) plug in via:

```csharp
Configure<AbpApiProxyScriptingOptions>(options =>
{
    options.Generators["my-generator"] = typeof(MyGenerator);
});
```

`AbpApiProxyScriptingOptions` is just a dictionary:

```csharp title="framework/src/Volo.Abp.Http/Volo/Abp/Http/ProxyScripting/Configuration/AbpApiProxyScriptingOptions.cs"
public class AbpApiProxyScriptingOptions
{
    public IDictionary<string, Type> Generators { get; }

    public AbpApiProxyScriptingOptions()
    {
        Generators = new Dictionary<string, Type>();
    }
}
```

`ProxyScriptManager.GetScript` is the entrypoint MVC views call (usually through a tag helper). It pulls a model from `IApiDescriptionModelProvider`, filters it by the requested modules/controllers/actions, and asks the chosen generator to render it:

```csharp title="framework/src/Volo.Abp.Http/Volo/Abp/Http/ProxyScripting/ProxyScriptManager.cs"
private string CreateScript(ProxyScriptingModel scriptingModel)
{
    var apiModel = _modelProvider.CreateApiModel(new ApplicationApiDescriptionModelRequestDto { IncludeTypes = false });

    if (scriptingModel.IsPartialRequest())
    {
        apiModel = apiModel.CreateSubModel(scriptingModel.Modules, scriptingModel.Controllers, scriptingModel.Actions);
    }

    var generatorType = _options.Generators.GetOrDefault(scriptingModel.GeneratorType);
    if (generatorType == null)
    {
        throw new AbpException($"Could not find a proxy script generator with given name: {scriptingModel.GeneratorType}");
    }

    using (var scope = _serviceProvider.CreateScope())
    {
        return scope.ServiceProvider
            .GetRequiredService(generatorType)
            .As<IProxyScriptGenerator>()
            .CreateScript(apiModel);
    }
}
```

The jQuery generator emits one IIFE per module, one nested IIFE per controller, and one function per action, each ending with `return abp.ajax(...)`:

```csharp title="framework/src/Volo.Abp.Http/Volo/Abp/Http/ProxyScripting/Generators/JQuery/JQueryProxyScriptGenerator.cs"
public const string Name = "jquery";

public string CreateScript(ApplicationApiDescriptionModel model)
{
    var script = new StringBuilder();
    script.AppendLine("/* This file is automatically generated by ABP framework to use MVC Controllers from javascript. */");
    ...
    AddInitializedEventTrigger(script);
    return script.ToString();
}

private static void AddInitializedEventTrigger(StringBuilder script)
{
    script.AppendLine("abp.event.trigger('abp.serviceProxyScriptInitialized');");
}
```

So a browser page that includes the generated script will receive an `abp.serviceProxyScriptInitialized` event once all controllers have been wired up — a convenient hook for UI bootstrap code.

## Parameter binding source IDs

`ParameterBindingSources` exposes the string IDs ABP uses internally (`"Path"`, `"Query"`, `"Header"`, `"Form"`, `"ModelBinding"`, ...). These match the `BindingSourceId` field on `MethodParameterApiDescriptionModel` and are how the URL/payload builders decide where each parameter goes.

```csharp
var versionParam = action.Parameters.FirstOrDefault(p =>
    p.Name == "apiVersion" && p.BindingSourceId == ParameterBindingSources.Path) ??
    action.Parameters.FirstOrDefault(p =>
        p.Name == "api-version" && p.BindingSourceId == ParameterBindingSources.Query);
```

The snippet (extracted from `JQueryProxyScriptGenerator`) is the same pattern used by `ClientProxyBase` to locate the API version parameter — see [dynamic-c-sharp-proxies](/http/dynamic-c-sharp-proxies).
